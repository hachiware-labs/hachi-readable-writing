# Relay Runner v1 から v2 への設定移行ガイド

この手順は、Relay Runner を停止リスクを抑えながら v1 から v2 へ段階的に切り替えるための運用ガイドです。まず v2 用の設定を作成・検証し、検証環境、10% の本番トラフィック、全体展開の順で進めます。

## 変更点

| v1 | v2 | 移行時の扱い |
|---|---|---|
| `workers` | `concurrency` | キー名のみ変更。意味と値は同じ。 |
| `retry.count` | `retry.max_attempts` | 値の意味が変わる。v1 の値に 1 を加える。 |
| `retry.delay_seconds` | `retry.initial_delay` | キー名を変更し、秒数を時間文字列へ変換する。 |
| なし | `queue` | v2 では必須。今回の移行では `default` を指定する。 |

v2 は不明な設定キーがあると起動に失敗します。v1 では警告にとどまっていたキーも、v2 用の設定ファイルからは削除または v2 の正式なキー名へ置き換えてください。

## 設定の変換例

次の v1 設定を例にします。

```yaml
workers: 8

retry:
  count: 3
  delay_seconds: 10
```

v2 では次のように変換します。

```yaml
concurrency: 8
queue: default

retry:
  max_attempts: 4
  initial_delay: 10s
```

この例では、v1 の `retry.count: 3` は「初回実行後に最大 3 回再試行する」設定です。v2 では初回も試行回数に含めるため、同じ動作にするには `retry.max_attempts: 4` とします。

## 移行手順

1. v1 の設定ファイルと v1 実行物を、ロールバック用として保持します。v2 用の設定を上書き保存する場合も、v1 に戻すための組み合わせを明確に残してください。

2. v1 設定を複製し、v2 用の設定ファイルを作成します。`workers`、`retry.count`、`retry.delay_seconds` を上記の対応に従って変換し、`queue: default` を追加します。

3. v2 用設定から不明なキーを取り除きます。ここに記載のない設定キーの v2 での扱いは不明です。残す場合は、v2 で有効なキーであることを別途確認してください。

4. 作成した設定を検証します。

   ```sh
   relay validate --config relay.yml
   ```

   `configuration valid` と表示された場合に、設定検証は成功です。表示されない場合は、本番切替を行わず、設定内容を修正してから再検証します。

5. 検証環境で v2 を 1 日間運用します。この間、エラー率と処理時間を確認します。

6. 問題がなければ、本番処理の 10% を v2 に切り替えます。切替後は 2 時間、エラー率と処理時間を継続して確認します。

7. 2 時間の確認で問題がなければ、対象を全体へ広げます。エラー率と処理時間の判断基準は、この切替を始める前に運用チームの基準として定めておきます。本ガイドでは数値基準は定義しません。

## ロールバック

切替後に問題が見つかった場合は、v1 設定ファイルと v1 実行物を同時に戻します。

v2 用設定を v1 実行物へそのまま渡してはいけません。`concurrency`、`queue`、`retry.max_attempts`、`retry.initial_delay` を含む v2 設定は、v1 の設定形式ではありません。ロールバックでは、保持しておいた v1 の設定ファイルと v1 実行物の組み合わせを使用します。
