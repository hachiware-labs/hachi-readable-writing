# Relay Runner v1 から v2 への設定移行ガイド

この手順は、Relay Runner を停止リスクを抑えながら v1 から v2 へ段階的に切り替えるためのものです。v2 は不明な設定キーがあると起動に失敗するため、切り替え前に設定を変換し、必ず検証します。

## 設定変更の要点

| v1 | v2 | 変更内容 |
|---|---|---|
| `workers` | `concurrency` | 名前のみ変更。意味は同じです。 |
| `retry.count` | `retry.max_attempts` | 初回を含む総試行回数になります。v1 の値に 1 を加えます。 |
| `retry.delay_seconds` | `retry.initial_delay` | 秒数ではなく、`10s` のような時間文字列で指定します。 |
| なし | `queue` | v2 では必須です。今回の移行では `default` を指定します。 |

v1 では不明な設定キーは警告にとどまりますが、v2 では起動に失敗します。ここに記載のない既存キーは、v2 で使用できるかを確認し、不明なまま v2 設定へ残さないでください。

## 変更前後の設定例

以下は `workers: 8`、再試行回数を初回以外で 3 回、再試行間隔を 10 秒としていた場合の例です。

### v1 の設定例

```yaml
workers: 8

retry:
  count: 3
  delay_seconds: 10
```

### v2 の設定例

```yaml
concurrency: 8
queue: default

retry:
  max_attempts: 4
  initial_delay: 10s
```

`retry.count: 3` は「初回の処理に加えて 3 回再試行する」設定です。v2 では初回を含む回数を指定するため、`retry.max_attempts: 4` に変換します。

## 段階移行の手順

1. v1 の設定ファイルと v1 実行物を、ロールバックに使える状態で保持します。v2 への切り替え後も、v1 に戻す際は設定ファイルと実行物を組み合わせて戻します。

2. v1 設定を v2 形式へ変換します。`workers`、`retry.count`、`retry.delay_seconds` を上記の対応に従って置き換え、`queue: default` を追加します。

3. 変換後の設定に、v2 で認識されないキーが残っていないことを確認します。v2 は不明なキーがあると起動できません。

4. 次のコマンドで v2 設定を検証します。

   ```text
   relay validate --config relay.yml
   ```

   `configuration valid` と表示された場合、設定検証は成功です。成功しない場合は、本番への切り替えを進めず、設定内容を修正して再検証します。

5. 検証環境で v2 を 1 日運用します。この期間は、v2 のエラー率と処理時間を確認します。

6. 本番トラフィックの 10% を v2 へ切り替えます。切り替え後 2 時間、エラー率と処理時間を確認します。

7. 2 時間の確認で問題がなければ、v2 の対象を全体へ広げます。

## ロールバック

エラー率または処理時間に問題がある場合は、v1 設定ファイルと v1 実行物を同時に戻します。v2 用に変換した設定ファイルを、そのまま v1 に渡してはいけません。
