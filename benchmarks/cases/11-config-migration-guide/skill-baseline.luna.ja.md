# Relay Runner v1からv2への設定移行ガイド

このガイドは、Relay Runnerの設定をv1からv2へ変換し、検証環境と本番環境で段階的に切り替えるための手順です。

## 設定の変換ルール

| v1 | v2 | 変換方法 |
|---|---|---|
| `workers` | `concurrency` | 値をそのまま移す |
| `retry.count` | `retry.max_attempts` | v1の値に1を加える |
| `retry.delay_seconds` | `retry.initial_delay` | 秒数を時間文字列に変換する |
| なし | `queue` | `queue: default`を追加する |

`retry.count`は初回を除く再試行回数です。一方、`retry.max_attempts`は初回を含む総試行回数です。そのため、変換式は次のとおりです。

```text
retry.max_attempts = retry.count + 1
```

たとえば、v1の`retry.count: 3`は、v2では`retry.max_attempts: 4`にします。

v2では不明な設定キーがあると起動に失敗します。v1で警告だけだったキーも、v2設定へそのまま残さないでください。ここに記載していない設定キーについては、v2での対応状況や変換方法が不明なため、受け付け可否を確認できたものだけを移行対象にします。

## YAMLの変更例

以下は、今回確認できている設定項目だけを使った最小構成の例です。

### 変更前：v1

```yaml
workers: 4
retry:
  count: 3
  delay_seconds: 10
```

### 変更後：v2

```yaml
concurrency: 4
retry:
  max_attempts: 4
  initial_delay: "10s"
queue: default
```

`workers`の値は`concurrency`へそのまま移します。`delay_seconds: 10`は、v2では時間文字列の`"10s"`に変換します。v2では、旧キーの`workers`、`retry.count`、`retry.delay_seconds`を残しません。

## 段階的な移行手順

### 1. v1の復旧に必要なものを保管する

移行前に、現在使用しているv1設定ファイルとv1実行物を、同じ組み合わせで復旧できるよう保管します。

ロールバックでは設定ファイルだけでなく、v1実行物も同時に戻します。

### 2. v2設定を作成する

v1設定をコピーし、次の変換を行います。

- `workers`を`concurrency`へ変更する
- `retry.count`の値に1を加え、`retry.max_attempts`へ変更する
- `retry.delay_seconds`を時間文字列にして、`retry.initial_delay`へ変更する
- `queue: default`を追加する
- v2で対応状況が確認できない設定キーを持ち込まない

検証対象のv2設定ファイルを、ここでは`relay.yml`とします。

### 3. v2設定を検証する

次のコマンドを実行します。

```text
relay validate --config relay.yml
```

成功時は、次の表示になります。

```text
configuration valid
```

この表示にならない場合は、v2実行物を起動せず、設定内容を修正してから再度検証します。

### 4. 検証環境で1日運用する

検証に成功したv2設定とv2実行物を検証環境へ適用し、1日動かします。

この期間に、設定変換による起動失敗や処理上の異常がないことを確認します。問題があれば本番へ進めず、v2設定を修正して再検証します。

### 5. 本番の10%をv2へ切り替える

検証環境での1日運用に問題がなければ、本番の10%をv2へ切り替えます。

このとき、v2へ切り替える対象には、変換済みのv2設定とv2実行物を組み合わせて適用します。残りの本番環境はv1のまま運用します。

### 6. 2時間確認してから全体へ広げる

10%を切り替えた状態で2時間運用し、次の2項目を確認します。

- エラー率
- 処理時間

このガイドでは具体的な許容値を定めていないため、既存の運用基準と比較して判断します。問題がなければ、残りの本番環境もv2へ切り替えます。異常があれば切り替えを拡大せず、ロールバックします。

## ロールバック

ロールバックする場合は、まずv2への切り替え拡大を止めます。その後、v1設定ファイルとv1実行物を同時に戻し、既存のv1運用手順で再開します。

v2設定をv1実行物へそのまま渡してはいけません。v2ではキー名や設定の意味が変更されているため、必ず対応するv1設定ファイルとv1実行物を組み合わせて使用します。

ロールバック後は、エラー率と処理時間を確認し、サービスが通常の状態へ戻ったことを確認してから移行作業を中断します。
