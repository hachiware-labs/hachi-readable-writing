# Relay Runner v1からv2への設定移行ガイド

本ガイドは、Relay Runnerをv1からv2へ段階的に切り替える運用担当者向けです。設定を変換して検証環境で安定性を確認した後、本番トラフィックの10%から移行し、問題がなければ全体へ広げます。

## 移行前の準備

ロールバックに備え、現在使用しているv1設定ファイルとv1実行物を保持してください。ロールバックでは両方を同時に戻す必要があります。

また、現在の設定に本ガイドで扱っていないキーが含まれる場合は、v2で使用できるかを事前に確認してください。v1は不明なキーがあっても警告にとどまりますが、v2は起動に失敗します。本ガイドに記載のないキーについては、変換方法を特定できません。

## 設定の変更点

| v1 | v2 | 変換方法 |
|---|---|---|
| `workers` | `concurrency` | キー名だけを変更し、値は維持する |
| `retry.count` | `retry.max_attempts` | v1の値に1を加える |
| `retry.delay_seconds` | `retry.initial_delay` | 秒数を時間文字列へ変換する |
| なし | `queue` | 必須項目として `default` を設定する |

`retry.count`と`retry.max_attempts`は数え方が異なります。v1の`count`は初回を除いた再試行回数ですが、v2の`max_attempts`は初回を含む総試行回数です。そのため、次の式で変換します。

```text
v2 retry.max_attempts = v1 retry.count + 1
```

たとえば、v1の`retry.count`が`3`なら、初回と3回の再試行を合わせて、v2の`retry.max_attempts`は`4`です。

## YAMLの変換例

以下は、`workers: 4`、再試行回数が3回、遅延が10秒だった場合の例です。数値は説明用です。

変更前のv1設定:

```yaml
workers: 4

retry:
  count: 3
  delay_seconds: 10
```

変更後のv2設定:

```yaml
concurrency: 4
queue: default

retry:
  max_attempts: 4
  initial_delay: 10s
```

この例では、同時実行数の意味を変えずに`workers`を`concurrency`へ改名しています。遅延時間は数値の`10`から時間文字列の`10s`へ変更しています。

## 段階的な移行手順

### 1. v2設定へ変換する

v1設定を複製し、v2用の設定として次の変更を行います。

1. `workers`を`concurrency`へ改名し、値を維持する。
2. `retry.count`を`retry.max_attempts`へ改名し、値に1を加える。
3. `retry.delay_seconds`を`retry.initial_delay`へ改名し、秒数を`10s`のような時間文字列にする。
4. `queue: default`を追加する。
5. v1用の古いキーが残っていないか確認する。

v2では不明なキーが起動失敗の原因になります。変換方法が分からないキーは推測で書き換えず、v2での扱いを確認してから移行を進めてください。

### 2. 設定を検証する

変換した設定を次のコマンドで検証します。

```console
relay validate --config relay.yml
```

成功時には、次のメッセージが表示されます。

```text
configuration valid
```

このメッセージが表示されない場合は、検証環境へ進まず、キー名、値の形式、必須の`queue`、および不明なキーの有無を確認してください。

### 3. 検証環境で1日稼働させる

検証に成功したv2設定を検証環境へ反映し、1日稼働させます。設定検証だけでは実際の処理結果までは確認できないため、この期間に運用上の問題がないことを確かめます。

問題が発生した場合は本番移行を開始せず、原因を確認して設定を修正し、再度検証してください。

### 4. 本番の10%をv2へ切り替える

検証環境で1日問題なく稼働した後、本番処理の10%をv2へ切り替えます。残りの90%はv1で稼働させ、影響を限定した状態で比較できるようにします。

切り替え後は、エラー率と処理時間を2時間確認します。

### 5. 全体へ広げる

2時間の確認でエラー率と処理時間に問題がなければ、v2の対象を本番全体へ広げます。

問題がある場合は全体移行を中止し、v1へロールバックします。許容できるエラー率や処理時間の基準値は本ガイドでは定められていないため、移行前に既存の運用基準を確認してください。

## ロールバック

ロールバック時は、v1設定ファイルとv1実行物を同時に戻します。

v2設定には、v1が認識しない`concurrency`、`retry.max_attempts`、`retry.initial_delay`、`queue`が含まれます。そのため、v2設定をそのままv1実行物へ渡してはいけません。必ず保持しておいたv1設定ファイルを使用してください。

ロールバック後は、v1実行物がv1設定を読み込んでいることを確認し、エラー率と処理時間が移行前の状態へ戻ったかを確認します。

## 移行完了の条件

次の状態になった時点で、v2への移行を完了とします。

- v2設定の検証結果が`configuration valid`である。
- 検証環境で1日、問題なく稼働している。
- 本番の10%を切り替えた後、エラー率と処理時間を2時間確認している。
- 確認結果に問題がなく、本番全体がv2へ切り替わっている。
- 必要な場合に備えて、v1設定ファイルとv1実行物を組み合わせて戻せる状態になっている。
