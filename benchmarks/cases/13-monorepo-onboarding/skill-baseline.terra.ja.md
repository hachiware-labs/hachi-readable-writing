# Northstar 開発者向け: 最初の変更を提出するまで

Northstar は、画面や API に業務ルールを重ねて実装するのではなく、共通の業務ルールを `packages/domain` に置く TypeScript モノレポです。最初の変更では、期限表示の判定を `packages/domain` で修正し、同じ場所にテストを追加します。

## まず開発環境をそろえる

Node.js 22 と pnpm 10 を用意し、取得したリポジトリのルートで依存関係をインストールします。

```sh
pnpm install --frozen-lockfile
```

このオプションでは、ロックファイルに記録された依存関係と一致しない状態でのインストールを防げます。依存関係を変更しない限り、ロックファイルは更新しません。

主なディレクトリは次のとおりです。

| 場所 | 役割 |
|---|---|
| `apps/web` | React による画面 |
| `apps/api` | HTTP API |
| `packages/domain` | 業務ルール |
| `packages/ui` | 共有画面部品 |

ローカルで画面と API を同時に起動する場合は、ルートで次を実行します。

```sh
pnpm dev
```

必要な環境変数や使用するポート番号は、この案内では未定です。起動時に不足が示された場合は、リポジトリ内の既存の案内やチームの設定を確認してください。

## 最初の課題を理解する

今回の課題は、`packages/domain` にある期限表示関数が、期限当日には `due today` を返すように修正することです。あわせて、その振る舞いを確認する単体テストを追加します。

判定は業務ルールなので、`apps/web` や `apps/api` に同じ条件分岐を追加しません。画面や API が期限表示を必要とする場合も、`packages/domain` の関数を利用する形を保ちます。

## 変更する

1. `packages/domain` で、期限表示関数と既存のテストを探します。まず現在の戻り値と、期限前・期限後・期限当日の扱いを確認します。

2. 期限当日の期待値を `due today` とする単体テストを追加します。既存のテストの書き方、日付の作り方、タイムゾーンの扱いに合わせてください。

3. 実装を修正し、期限当日にその関数が `due today` を返すようにします。期限前・期限後など、既存の振る舞いを意図せず変えないようにします。

日付の境界は不具合が入りやすい箇所です。テストでは「現在時刻」ではなく、既存の設計に従って比較基準となる日付を明確に与えられるなら、その形を優先します。

## 確認する

変更後は、まず対象パッケージだけを確認します。`<package-name>` には `packages/domain` の `package.json` に定義されたパッケージ名を指定します。

```sh
pnpm --filter <package-name> test
```

次に、リポジトリ全体の型検査と整形確認を行います。

```sh
pnpm typecheck
pnpm format:check
```

提出前には、全体テストも実行します。

```sh
pnpm test
```

すべて成功したことを確認し、変更内容には期限当日のテストと `packages/domain` の修正だけが含まれていることを見直します。依存関係を変更していない場合は、ロックファイルを含めません。

ブランチ名や提出方法はこの案内では定められていません。チームで案内されている手順に従って、修正内容、追加したテスト、実行した確認コマンドを添えて提出してください。
