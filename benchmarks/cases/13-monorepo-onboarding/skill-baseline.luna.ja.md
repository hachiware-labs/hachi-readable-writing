# Northstar 開発参加ガイド

このガイドでは、TypeScriptモノレポ「Northstar」を取得したあと、`packages/domain` の小さな修正とテスト追加を行い、レビューに提出できる状態まで進めます。

## 1. 開発環境を確認する

Northstarでは、次のバージョンを使います。

- Node.js 22
- pnpm 10

ターミナルでバージョンを確認します。

```bash
node --version
pnpm --version
```

Node.jsが22系、pnpmが10系であることを確認してください。

## 2. 依存関係をインストールする

リポジトリのルートディレクトリへ移動し、ロックファイルに記録された内容どおりに依存関係をインストールします。

```bash
pnpm install --frozen-lockfile
```

`--frozen-lockfile` を付けることで、意図せず依存関係やロックファイルが変更されるのを防げます。ロックファイルは、依存関係を変更するときだけ更新してください。

## 3. モノレポの構成を確認する

Northstarの主なパッケージは次のとおりです。

- `apps/web`: React画面
- `apps/api`: HTTP API
- `packages/domain`: 業務ルール
- `packages/ui`: 画面で共有する部品

今回の期限表示の判定は、画面やAPIではなく `packages/domain` に実装します。`apps/web` や `apps/api` に同じ判定処理を複製しないでください。業務ルールを `packages/domain` に集約すると、利用箇所が増えても判定のずれを防げます。

## 4. 開発サーバーを起動する

Web画面とAPIを同時に起動するには、リポジトリのルートで次を実行します。

```bash
pnpm dev
```

ポート番号はこの案内では指定していないため、起動時の表示やプロジェクト内の設定を確認してください。画面確認が不要な場合は、開発サーバーを起動せずにテストと型検査を進めても構いません。

## 5. 最初の課題を確認する

最初の課題は、`packages/domain` にある期限表示関数を修正し、期限当日の表示を `due today` にすることです。

まず、`packages/domain` 内で期限表示関数と既存テストを探します。関数名やファイル構成はリポジトリの実装に従ってください。

修正するときは、次の方針を守ります。

- 期限日が当日なら `due today` を返す。
- 既存の期限切れや未来の日付に対する表示は、現在の仕様を維持する。
- 判定処理は `packages/domain` に置く。
- `apps/web` や `apps/api` に同じ日付判定を追加しない。

既存テストの書き方と日付の扱いを確認したうえで、期限当日のケースを単体テストに追加します。テストでは、期限日と判定時点の日付が同じ場合に、結果が正確に `due today` になることを検証してください。

## 6. 対象パッケージのテストを実行する

`packages/domain` の `package.json` にある `name` の値を確認し、そのパッケージ名を使ってテストを実行します。

```bash
pnpm --filter <package-name> test
```

`<package-name>` は、`packages/domain/package.json` の `name` に置き換えてください。

テストが失敗した場合は、まず次を確認します。

- 期限日と判定時点の日付を同じ条件で比較しているか。
- タイムゾーンや時刻まで比較していないか。
- 既存の期限切れ・未来の日付の挙動を壊していないか。
- テストが実装の詳細ではなく、期限表示の結果を検証しているか。

## 7. 変更全体を検証する

対象パッケージのテストが通ったら、ルートで次を順番に実行します。

```bash
pnpm typecheck
pnpm format:check
pnpm test
```

提出前には全体テストも必ず実行してください。型検査、整形確認、全体テストのいずれかが失敗した場合は、原因を修正してから再実行します。

## 8. 差分を確認して提出する

検証が終わったら、変更内容を確認します。

```bash
git status
git diff
```

今回の差分に、次の内容だけが含まれていることを確認してください。

- `packages/domain` の期限表示関数の修正
- 期限当日の単体テスト
- 必要な場合に限るテスト関連の変更

依存関係を変更していない場合、ロックファイルは変更しません。意図しないファイルや生成物が差分に含まれていないことも確認します。

チームで決められた作業ブランチ名を使い、変更をコミットして通常のレビュー方法で提出してください。提出時には、次の点を簡潔に記載します。

- 期限当日の表示を `due today` に修正したこと
- `packages/domain` に単体テストを追加したこと
- 対象パッケージのテスト、`pnpm typecheck`、`pnpm format:check`、`pnpm test` が通っていること
