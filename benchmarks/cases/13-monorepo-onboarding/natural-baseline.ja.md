# Northstar 開発スタートガイド

この案内では、リポジトリを取得した後のセットアップから、最初の小さな変更を提出するまでを説明します。

## 1. 必要なツールを確認する

Northstarでは次のバージョンを使います。

- Node.js 22
- pnpm 10

```sh
node --version
pnpm --version
```

Node.jsが `v22.x`、pnpmが `10.x` であることを確認してください。

## 2. 依存関係をインストールする

リポジトリのルートへ移動し、ロックファイルどおりに依存関係をインストールします。

```sh
pnpm install --frozen-lockfile
```

`--frozen-lockfile` を付けることで、意図せずロックファイルが書き換わるのを防げます。

## 3. モノレポの構成を把握する

主なディレクトリは次のとおりです。

| ディレクトリ | 役割 |
|---|---|
| `apps/web` | Reactで実装された画面 |
| `apps/api` | HTTP API |
| `packages/domain` | 業務ルールや判定処理 |
| `packages/ui` | 複数画面で共有するUI部品 |

業務上の判定は、原則として `packages/domain` に置きます。`apps/web` や `apps/api` に同じロジックを複製しないでください。

## 4. 開発環境を起動する

リポジトリのルートで次を実行します。

```sh
pnpm dev
```

webとapiが同時に起動します。

環境変数やポート番号は、この案内の時点では未確認です。起動時に不足を示すエラーが出た場合は、リポジトリ内のREADMEや環境変数のサンプルを確認し、それでも分からなければチームに確認してください。値を推測して設定しないようにします。

## 5. 最初の課題に取り組む

最初の変更は、`packages/domain` にある期限表示関数を修正し、期限当日に次の文字列を返すようにすることです。

```text
due today
```

まず、既存の期限表示関数とテストを探します。たとえば、リポジトリのルートから次のように検索できます。

```sh
rg "due today|due date|deadline" packages/domain
```

関数名が分かっている場合は、その名前で検索してください。

実装時には次の点を確認します。

- 期限当日を正しく判定する
- 既存の期限前・期限後の表示を壊さない
- 日付の比較方法やタイムゾーンについて、既存コードの規約に従う
- 判定処理は `packages/domain` に置く
- `apps/web` や `apps/api` に同じ処理を追加しない

既存のテストファイルに、少なくとも「期限当日は `due today` を返す」ケースを追加します。可能であれば、既存の期限前・期限後のケースも確認して境界条件が明確になるようにしてください。

## 6. 変更を検証する

最初に、変更したパッケージだけをテストします。

```sh
pnpm --filter <package-name> test
```

`<package-name>` には、`packages/domain/package.json` の `name` を指定します。名前は次のコマンドなどで確認できます。

```sh
pnpm --filter ./packages/domain exec node -p "require('./package.json').name"
```

続いて、リポジトリ全体の型検査と整形確認を実行します。

```sh
pnpm typecheck
pnpm format:check
```

提出前には全体テストも実行します。

```sh
pnpm test
```

つまり、提出前の確認順は次のとおりです。

```sh
pnpm --filter <package-name> test
pnpm typecheck
pnpm format:check
pnpm test
```

## 7. 差分を確認する

コミットする前に、変更範囲を確認します。

```sh
git status
git diff
```

特にロックファイルが変更されていないか確認してください。今回の課題では依存関係を追加・更新する必要がないため、通常はロックファイルを変更しません。ロックファイルを更新するのは、依存関係を意図的に変更した場合だけです。

## 8. 変更を提出する

ブランチ名の規則はまだ案内されていません。チームの規約や既存のプルリクエストを確認してからブランチを作成してください。

変更をコミットし、チームで利用している方法でプルリクエストを作成します。説明には次の内容を簡潔に含めます。

- 期限当日の表示を `due today` に変更したこと
- 期限当日の単体テストを追加したこと
- 判定ロジックを `packages/domain` に維持したこと
- 実行した検証コマンドと結果

提出前の最終チェックです。

- 対象パッケージのテストが成功している
- 全体の型検査が成功している
- 整形確認が成功している
- 全体テストが成功している
- webやapiに判定処理を複製していない
- 不要なファイルを変更していない
- 依存関係を変えていない場合、ロックファイルを変更していない
- 未確認の環境変数、ポート番号、ブランチ名を推測していない
