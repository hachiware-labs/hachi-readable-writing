# Northstar 開発者向け：最初の変更を提出するまで

Northstar は pnpm ワークスペースで管理する TypeScript モノレポです。業務ルールは `packages/domain` に集約されているため、最初の課題もここを変更します。画面や API に同じ判定を複製しないことが重要です。

## 1. 開発環境を準備する

Node.js 22 と pnpm 10 を用意したうえで、リポジトリを取得してルートディレクトリで次を実行します。

```sh
pnpm install --frozen-lockfile
```

このコマンドは、既存のロックファイルに記録された依存関係をそのまま再現します。依存関係を追加・更新しない今回の変更では、ロックファイルを更新しません。

開発中に画面と API をまとめて起動したい場合は、ルートで次を実行します。

```sh
pnpm dev
```

環境変数、利用ポート、ブランチ名の運用は、現時点では別途案内が必要です。推測して設定せず、チームの案内を確認してください。

## 2. リポジトリの構成を把握する

主な担当範囲は次のとおりです。

- `apps/web`：React で作られた画面
- `apps/api`：HTTP API
- `packages/domain`：業務ルール
- `packages/ui`：共有画面部品

期限表示の判定は業務ルールなので、修正箇所は `packages/domain` です。`apps/web` や `apps/api` 側で同じ条件分岐を書かず、必要なら domain パッケージの関数を利用します。

## 3. 最初の課題を実装する

期限表示関数を探し、期限当日の値を受け取ったときに `due today` を返すよう修正します。既存の「期限切れ」「残り日数あり」などの扱いを確認してから、当日判定がどの条件の間に入るべきかを決めてください。

続けて、同じ `packages/domain` に単体テストを追加します。テストでは少なくとも、期限当日の入力が次を返すことを確認します。

```text
due today
```

日付を扱うテストでは、実行する時刻やタイムゾーンによって結果が変わらないよう、入力となる現在日時と期限日時を明示的に固定します。

## 4. 変更を確認する

まず、対象パッケージだけのテストをルートから実行します。

```sh
pnpm --filter <domain-package-name> test
```

`<domain-package-name>` には、`packages/domain/package.json` の `name` に定義されたワークスペース名を指定します。

その後、リポジトリ全体の型検査と整形確認を実行します。

```sh
pnpm typecheck
pnpm format:check
```

提出前には、全体テストも実行します。

```sh
pnpm test
```

すべて成功し、変更が `packages/domain` の実装とテストに収まっていること、依存関係を変更していないためロックファイルに不要な差分がないことを確認してから提出してください。
