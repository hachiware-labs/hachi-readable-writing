# Northstar 開発参加ガイド

この案内では、TypeScriptモノレポ「Northstar」を取得した開発者が、開発環境を準備し、最初の小さな変更を提出できる状態になるまでを説明します。

## 1. 開発環境を準備する

Northstarでは、次のバージョンを使用します。

- Node.js 22
- pnpm 10

バージョンを確認します。

```console
node --version
pnpm --version
```

Node.jsが `v22`、pnpmが `10` で始まることを確認してください。バージョン管理ツールの指定はまだ案内されていないため、必要に応じてチームへ確認します。

リポジトリのルートへ移動し、ロックファイルに記録された内容どおりに依存関係をインストールします。

```console
pnpm install --frozen-lockfile
```

`--frozen-lockfile`を付けることで、インストール時にロックファイルが意図せず変更されるのを防げます。

## 2. リポジトリの構成を理解する

最初に、変更内容をどこへ置くべきか把握しておきます。

| ディレクトリ | 役割 |
|---|---|
| `apps/web` | Reactで構築された画面 |
| `apps/api` | HTTP API |
| `packages/domain` | 業務ルール |
| `packages/ui` | 複数の画面で共有する部品 |

画面やAPIが業務上の同じ判断を必要とする場合、その判断は原則として`packages/domain`に置きます。`apps/web`や`apps/api`へ同じ処理を複製すると、修正漏れによって結果が食い違う可能性があるためです。

## 3. 現在の状態を確認する

WebアプリとAPIは、リポジトリのルートから同時に起動できます。

```console
pnpm dev
```

必要な環境変数と使用するポート番号は、まだ案内されていません。起動時に不足を示すエラーが出た場合は、値を推測せず、既存のREADMEやサンプル設定を確認したうえでチームへ問い合わせてください。

変更前の基準を確認するには、次の検査をルートで実行します。

```console
pnpm test
pnpm typecheck
pnpm format:check
```

最初から失敗する検査がある場合は、自分の変更によるものと混同しないよう、失敗したコマンドとメッセージを記録しておきます。

## 4. 最初の課題に取り組む

最初の変更は、`packages/domain`にある期限表示関数の修正です。期限が当日の場合に、関数が次の文字列を返すようにします。

```text
due today
```

まず`packages/domain`内で、期限表示関数と既存の単体テストを探します。実装を読むときは、期限前や期限超過など、当日以外の判定がどのように扱われているかも確認してください。

テストには少なくとも「期限が当日なら`due today`を返す」という事例を追加します。日付や時刻の扱いは既存実装とテストの方針に合わせ、今回の課題に必要のない仕様変更は加えません。

判定処理は`packages/domain`で完結させます。表示結果を利用するために`apps/web`や`apps/api`を変更する必要が生じても、同じ日付判定をそれぞれに書かないでください。

## 5. 変更を検証する

最初に、対象パッケージだけを検査して素早く問題を見つけます。

```console
pnpm --filter <package-name> test
```

`<package-name>`には、`packages/domain/package.json`の`name`に記載されたパッケージ名を指定します。ディレクトリ名から推測せず、実際の値を確認してください。

対象パッケージのテストが通ったら、ルートで型検査と整形確認を実行します。

```console
pnpm typecheck
pnpm format:check
```

提出前には全体テストも実行します。

```console
pnpm test
```

失敗した場合は、コマンド、主なエラー内容、自分の変更との関係を確認します。変更前から存在していた失敗を除き、追加したテストを無効化したり検査対象から外したりして提出しないでください。

## 6. 提出前に差分を確認する

最終的な差分に、期限表示関数の修正と対応する単体テストだけが含まれていることを確認します。

依存関係を変更していない場合、ロックファイルを更新する必要はありません。意図せず変更されていた場合は原因を確認してください。依存関係を追加、削除、更新した場合に限り、その変更に対応するロックファイルの更新を含めます。

提出内容には、次の情報を簡潔に記載します。

- 期限当日の表示を`due today`へ変更したこと
- 期限当日の単体テストを追加したこと
- 対象パッケージのテスト、全体の型検査、整形確認、全体テストの結果

ブランチ名の規則や提出方法はまだ案内されていません。命名や手順を独自に決めず、チームの運用を確認してから提出してください。
