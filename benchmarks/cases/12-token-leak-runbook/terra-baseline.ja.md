# 公開リポジトリへ誤コミットした本番APIトークンの対応手順

対象は、本番の読み書き権限を持つAPIトークンです。漏えいを確認した時点を **T0** とします。

## 1. 直ちに封じ込める

当番エンジニアは、認証基盤で対象トークンを失効させます。**新トークンの発行より先に失効**させてください。失効時刻を記録します。

トークン値は、記録・チケット・チャット・メールへ貼り付けません。識別が必要な場合は、認証基盤上のトークンIDなど、値そのものではない情報を用います。

## 2. 最小限の証跡を残す

失効後、リポジトリを非公開化したり履歴を書き換えたりする前に、次を記録します。

| 記録項目 | 内容 |
|---|---|
| コミットID | トークンを含むコミットを特定できるID |
| 発見時刻 | T0 |
| リポジトリURL | 公開されていた対象リポジトリ |
| 通知者 | 漏えいを報告した人または検知元 |
| 失効時刻 | 認証基盤で旧トークンを失効した時刻 |

## 3. サービスを復旧する

旧トークンの失効後に、新トークンを発行します。新トークンは秘密管理基盤へ保存し、利用サービスの参照先を更新して再起動します。サービスが新トークンで動作していることを確認します。

## 4. 影響を調査し、不審操作を修復する

失効時刻をT1として、**T0の24時間前からT1まで**の認証ログと操作ログを確認します。旧トークンによる以下を特定します。

- 送信元
- 実行時刻
- 実行された操作

不審な書き込みが見つかった場合は、変更対象を特定し、旧トークンとは別の正常な資格情報で修復します。

## 5. 公開履歴を処理する

リポジトリの非公開化または履歴からの秘密情報除去を実施します。履歴を書き換えても、公開済みのトークンを再び有効にしてはいけません。旧トークンは失効したままにします。

## 6. 連絡と判断

インシデント責任者は、次の関係者へ連絡します。

- セキュリティー担当
- サービス所有者
- 広報・法務判断の責任者

外部通知の要否は、インシデント責任者が判断します。

## 7. 終了条件

次のすべてを満たした時点で、本対応を終了します。

- 旧トークンを失効済みである
- 新トークンを秘密管理基盤へ保存し、利用サービスへ反映済みである
- 対象期間の認証ログ・操作ログを確認済みである
- 不審操作があれば、影響確認と修復を完了している
- 公開リポジトリの履歴または公開状態を処理済みである
- 事後レビューの担当者と実施期限を決定している
