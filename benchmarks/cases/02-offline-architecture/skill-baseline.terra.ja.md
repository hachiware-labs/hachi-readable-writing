# トンネル点検アプリのオフライン対応設計案

## 目的

トンネル内で最長8時間通信できない状況でも、点検員がAndroidタブレットで点検票の記入と写真撮影を継続できるようにする。通信が回復した後は、写真と点検データを安全に同期し、同一の点検票に対する後続の訂正を黙って上書きしない。

この設計では、端末での作業継続、写真を含む確実な再送、変更履歴の7年間保存を満たす一方、端末内状態と同期処理の複雑さ、写真の一時保存容量、競合解決の遅延を受け入れる。

## 全体方針

端末を、通信がなくても点検作業を完結できる一時的な作業場所とする。端末には暗号化したSQLiteを置き、点検票、写真の参照情報、未送信操作を保存する。写真本体は端末の暗号化領域にファイルとして保存し、SQLiteにはファイルパスとチェックサムを記録する。

通信が回復すると、端末は写真を先にオブジェクトストレージへ送信し、成功を確認してから点検票などのメタデータ操作をサーバーへ送る。サーバーは点検票の版番号を使って更新競合を判定し、競合した変更を保留して監督者の確認キューへ送る。

```text
Androidタブレット
  ├─ 暗号化SQLite
  │    ├─ 点検票のローカル状態
  │    ├─ 写真のパス・チェックサム
  │    └─ 未送信操作（outbox）
  └─ 暗号化領域の写真ファイル
          │
          │ 通信回復後
          ▼
オブジェクトストレージ ← 写真を先に送信
          │
          ▼
既存CRUD API
  ├─ 点検票の版番号による更新判定
  ├─ 競合の保留と監督者確認キュー
  └─ 追記専用の変更イベント保存
```

## 設計判断

### 端末をオフライン作業の正本とする

通信断中はサーバーを参照できないため、点検票の入力・修正と写真の登録は端末内で完結させる。点検員は通信状態によって操作可否が変わらないため、トンネル内でも通常どおり作業を続けられる。

端末内の点検票は同期済みの最新状態を基礎にし、オフライン中の変更をすぐ反映する。ただし、通信断中に他者が同じ点検票を更新している可能性は端末からは分からない。この不確実性は、同期時の版番号照合と監督者による競合確認で扱う。

### 写真とメタデータを分離して保存・同期する

写真は1台につき1日最大200枚撮影されるため、SQLiteのBLOBとして保持せず、端末の暗号化領域にファイルとして保存する。SQLiteには写真ファイルのパス、チェックサム、同期状態を持たせる。これにより、点検票の検索や更新と、大きな写真データの送受信を分離できる。

同期時は写真を先にチェックサム付きでオブジェクトストレージへ送る。写真の送信成功を確認できたものだけを参照するメタデータ操作をサーバーへ送るため、サーバー上の点検票が未到達の写真を指す状態を避けられる。

一方で、写真送信に成功した直後、メタデータ送信前に通信が切れることはあり得る。この場合は、写真送信済みの記録を端末に残し、次回同期でメタデータ送信を再開する。再送時に同じ写真を重複保存しないよう、チェックサムを写真の識別に用いる。

### outboxで未送信操作を管理する

点検票の作成、更新、写真の追加といった端末上の変更は、ローカル状態の更新と同時にoutboxへ記録する。outboxには少なくとも、操作の一意な識別子、対象となる点検票、作成時点での版番号、変更内容、関連写真のチェックサム、作成時刻を保存する。

同期処理はoutboxを順に処理し、成功した操作だけを完了として記録する。途中でアプリが停止した場合や通信が切れた場合でも、未完了の操作は次回に再実行できる。サーバー側も操作の一意な識別子を受け取り、同じ操作が再送されても同じ結果を返すようにする。これにより、通信失敗時に「処理済みか不明」な操作を安全に再送できる。

### 版番号で競合を検出し、監督者が解決する

点検票にはサーバー管理の版番号を持たせる。端末は更新時に、編集開始時点または直前に同期した版番号を添えて送信する。サーバー上の版番号と一致するときだけ更新を適用し、更新後に新しい版番号を返す。

版番号が一致しない場合、サーバーは変更を黙って上書きしない。端末から届いた変更を競合として保留し、監督者の確認キューへ送る。監督者は、現在のサーバー側の内容と保留された変更を比較し、採用または訂正を判断する。

通常は点検票を1人が編集するため、競合は例外的な経路になると見込む。ただし、後から監督者が訂正する運用があるため、競合を自動併合する前提にはしない。写真の追加、点検票の項目修正、削除などで扱いが異なる場合の具体的な確認画面や解決規則は、現時点では未決定である。

### 変更履歴は追記専用イベントとして保存する

サーバーは、点検票の現在状態とは別に、変更イベントを追記専用で保存する。各イベントには、変更者、変更時刻、変更元端末、操作識別子、対象点検票、変更前後の版番号、変更内容または参照情報を記録する。

この記録により、現在の内容だけでは分からない「誰が、いつ、どの端末で変更したか」を後から確認できる。変更イベントは7年間保存する対象とする。端末のoutboxは同期完了後の送信制御用であり、7年間の履歴保管を端末に依存しない。

### 端末紛失時の影響を限定する

端末に保存するSQLiteと写真ファイルは暗号化する。暗号化に用いる端末鍵はOSの安全な鍵領域に保存し、アプリや通常のファイル領域から直接取り出せないようにする。

管理者は端末を失効できるようにする。失効後の端末は、次回の通信時に同期や利用を継続できない状態にする。すでに通信断中の端末を即時に遠隔操作できるわけではないため、失効が端末へ反映されるまでの扱いと、端末再接続時のデータ回収可否は別途運用として決める必要がある。

## 同期の流れ

1. 端末は通信可能になった時点でoutboxから未完了操作を取得する。

2. 操作に関連する未送信写真があれば、チェックサムを添えてオブジェクトストレージへ送る。送信済みの同一チェックサムの写真は再利用し、重複保存しない。

3. 必要な写真がすべて送信済みになった後、点検票の作成・更新などのメタデータ操作を、操作識別子と基準版番号を添えてCRUD APIへ送る。

4. サーバーが操作を適用した場合、端末は新しい版番号と同期済み状態を保存し、outboxの操作を完了にする。

5. サーバーが版番号不一致を返した場合、端末はその操作を競合状態として保持する。サーバーは保留変更を監督者確認キューへ登録し、端末は利用者に同期完了ではないことを示す。

6. 一時的な通信失敗やアプリ停止では操作を完了にせず、次回同期で同じ操作を再実行する。

## データの責務

| データ | 端末での保持 | サーバーでの保持 | 主な目的 |
|---|---|---|---|
| 点検票 | 暗号化SQLite | 既存CRUD APIのデータ | オフライン編集と共有された現在状態 |
| 写真本体 | 暗号化領域のファイル | オブジェクトストレージ | 大容量データの保存 |
| 写真のパス・チェックサム | 暗号化SQLite | 点検票メタデータとして参照 | 写真の識別と送信確認 |
| 未送信操作 | 暗号化SQLiteのoutbox | 操作識別子の処理済み記録 | 安全な再送と重複防止 |
| 変更履歴 | 必要に応じたローカル参照のみ | 追記専用イベントストア | 7年間の監査証跡 |
| 競合中の変更 | 端末の操作状態 | 監督者確認キュー | 上書き防止と判断待ち |

## 代償と影響

この設計では、端末の状態と同期処理がオンライン専用のCRUDより複雑になる。特に、写真送信済み・メタデータ未送信・競合保留といった中間状態を正しく扱う必要がある。

また、最長8時間のオフライン作業と1日最大200枚の写真に対応するため、端末には写真の一時保存容量が必要になる。必要容量は写真1枚あたりの実際のサイズと保持期間に依存するため、現時点では算定できない。端末配布前に、想定画質、再送期間、容量不足時の利用者向け挙動を決める必要がある。

競合は即時に解決されず、監督者の確認を待つ。その代わり、監督者の訂正や端末間の編集が発生した場合でも、点検結果を黙って失うことを避けられる。

## 設計審査で決める事項

1. 競合時に監督者へ提示する差分と、採用・訂正・却下の操作をどこまで提供するか。
2. 写真の想定サイズ、端末に確保する一時保存容量、同期完了後の端末内写真の保持期間。
3. 端末失効後に再接続した端末をどのように扱うか。未送信データの回収を許可するか、利用停止を優先するか。
4. 既存CRUD APIに追加する、操作識別子による重複防止、版番号照合、競合保留、変更イベント記録の責務分担。
