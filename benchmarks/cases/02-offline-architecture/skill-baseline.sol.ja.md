# トンネル点検アプリ オフライン対応設計案

## 1. 目的と設計方針

本設計は、点検員がトンネル内で最長8時間通信できない状況でも、Androidタブレットで点検票の編集と写真撮影を継続できるようにするものである。通信復旧後は、端末に蓄積した変更を安全にサーバーへ反映し、競合や再送によってデータを失ったり重複させたりしないことを重視する。

対象規模は点検員40人、端末1台あたり1日最大200枚の写真である。点検票は通常1人が編集するが、同期前後に監督者が訂正する可能性があるため、単純な後勝ちの上書きは採用しない。また、変更履歴を7年間保存する必要がある。

設計の中心となる判断は次のとおりである。

- 端末を一時的に独立して動作できるデータ主体とし、点検中はローカル保存を完了条件とする。
- 同期にはoutbox方式を使い、再実行しても同じ変更や写真が重複しないようにする。
- 写真本体を先に保存し、参照可能になってからメタデータを反映する。
- 点検票の版番号が一致しない更新は上書きせず、競合として人が確認する。
- 業務データの現在値とは別に、変更イベントを追記専用で保存する。
- 端末内データを暗号化し、紛失端末を管理者が失効できるようにする。

## 2. 全体構成

端末では、点検票、写真の管理情報、同期状態、未送信操作を暗号化SQLiteに保存する。写真本体は端末の暗号化領域にファイルとして保存し、SQLiteにはファイルパスとチェックサムを記録する。

サーバー側では、現行のCRUD APIをそのまま端末から呼ぶのではなく、オフライン同期に必要な版管理、重複排除、競合保留を扱える同期処理を設ける。写真は既存のオブジェクトストレージへ保存し、点検票や写真メタデータは業務データとして保持する。これとは別に、すべての変更を追記専用イベントとして記録する。

```text
Androidタブレット
├─ 暗号化SQLite
│  ├─ 点検票と版番号
│  ├─ 写真のパス、チェックサム、送信状態
│  └─ outbox（未送信操作）
└─ 暗号化ファイル領域
   └─ 写真本体
          │
          │ 通信復旧後に同期
          ▼
同期処理
├─ 写真を先に送信 ──────────→ オブジェクトストレージ
├─ 点検票・写真メタデータ ──→ 業務データ
├─ 版不一致の更新 ──────────→ 競合確認キュー
└─ すべての変更 ───────────→ 追記専用イベント
```

## 3. 端末内での保存

点検員が点検票を更新したときは、点検データの更新とoutboxへの操作追加を同じSQLiteトランザクションで行う。これにより、画面上では保存済みなのに同期対象へ登録されていない状態を避ける。

写真撮影時は、写真本体を暗号化領域へ保存し、そのパスとチェックサムをSQLiteへ登録する。ファイル保存とSQLite更新を単一トランザクションにはできないため、途中失敗を検出できる状態管理が必要となる。たとえば「ファイル作成中」「ローカル保存済み」「送信済み」のように状態を持ち、起動時または同期前に、実体のないレコードと参照されていないファイルを検出する。ただし、具体的な状態名や回復手順は実装設計で確定する。

ローカル保存が成功した時点で、点検員には端末内への保存完了を示す。一方、サーバーへの反映完了とは区別し、未同期、同期中、競合、同期済みを判別できるようにする必要がある。画面上の表現や通知方法は、本設計では未決定とする。

## 4. 同期処理

同期処理は通信の中断を通常の事象として扱う。処理の途中で通信が切れても、次回は未完了の処理から再開し、同じ要求が複数回届いても結果が重複しないようにする。

各outbox操作には、端末で生成した一意な操作IDを付けることを推奨する。サーバーは処理済み操作IDを識別し、再送された場合には新しい変更として適用せず、以前の処理結果を返す。写真についてはチェックサムを使って、送信内容の破損検出と同一データの識別を行う。チェックサムだけで写真の業務上の同一性を判断するか、写真IDと組み合わせるかは詳細設計で決める。

同期順序は次のとおりとする。

1. 端末は写真本体をチェックサム付きでオブジェクトストレージへ送信する。
2. 保存成功を確認した後、写真の参照情報を含むメタデータをサーバーへ送る。
3. サーバーは点検票の版番号を確認し、適用、競合保留、または再送済みのいずれかとして処理する。
4. 端末はサーバーの受理結果を保存してから、対応するoutbox操作を完了扱いにする。
5. サーバー側の新しい状態や監督者による訂正を端末へ取り込み、ローカル状態を更新する。

この順序により、メタデータだけが登録され、参照先の写真が存在しない状態を避ける。写真送信後にメタデータ送信が失敗した場合は、写真を再送または再利用して処理を続ける。未参照の写真をいつ削除するかは、保持期間や運用上の復旧要件と合わせて別途決める必要がある。

同期対象が多数ある場合の送信順、並列数、再試行間隔、モバイル回線での送信可否は未決定である。これらは写真サイズ、端末容量、利用可能な通信条件を測定したうえで設定する。

## 5. 点検票の版管理と競合

サーバーは点検票ごとに版番号を持つ。端末は編集を始めた時点の版番号を保持し、更新内容とともに送信する。

送信された版番号がサーバーの現在版と一致する場合、サーバーは変更を適用して版番号を進める。一致しない場合は、端末側の変更で現在値を黙って上書きせず、競合として保留し、監督者の確認キューへ送る。

```text
端末が保持する版 = サーバーの現在版
    └─ 変更を適用し、新しい版を発行

端末が保持する版 ≠ サーバーの現在版
    └─ 変更を保留し、競合確認キューへ送る
```

競合レコードには、少なくともサーバーの現在値、端末から届いた変更、変更元の版、操作者、操作時刻、端末識別情報を関連付ける。監督者がどの単位で差分を確認し、片方を採用するのか、項目単位で統合できるのかは、業務要件を確認して決定する。

競合解決の結果も新たな変更として扱い、元の履歴を書き換えない。これにより、端末で行われた変更、競合が発生した事実、監督者が確定した内容を後から追跡できる。

## 6. 変更履歴

サーバーでは、業務データの現在値とは別に変更イベントを追記専用で保存する。各イベントには、誰が、いつ、どの端末で、どの点検票に対して変更したかを記録する。監督者による競合解決や訂正も同じ履歴の対象とする。

変更履歴は7年間保存する。追記済みイベントを通常の更新処理で変更または削除できないようにし、現在値の更新に失敗した場合と履歴記録に失敗した場合が分離しない構成が必要である。業務データとイベントを同じデータベースで管理する場合は同一トランザクションで記録できる。異なる保存先を使う場合は、欠落を回復できる永続キューなどが必要になる。

操作時刻については、端末が記録した時刻とサーバーが受信した時刻を分けて保存することを推奨する。長時間オフラインでは両者が大きく異なるためである。端末時計のずれをどのように扱うか、履歴の検索・出力要件、7年経過後の削除方法は未決定事項として残る。

## 7. 端末紛失への対応

SQLite、写真、同期用の一時データは端末上で暗号化する。暗号鍵はアプリ内や通常のファイル領域には保存せず、Android OSの安全な鍵領域へ格納する。

管理者は端末を失効できるものとし、失効済み端末からの同期要求をサーバーが拒否する。失効情報を受け取ったアプリがローカルデータを利用不能にする処理も検討対象となるが、端末がオフラインの間はサーバーから失効を通知できない。このため、遠隔失効だけで紛失直後のローカル閲覧を完全に防げるとは限らず、端末ロックやOSの管理機能と組み合わせる必要がある。

端末の認証方式、失効確認の頻度、オフラインで利用を継続できる認証期間、鍵を失った場合の復旧可否は、セキュリティ要件として別途決定する。

## 8. 現行システムへの影響

現行のオンライン前提CRUD APIには、少なくとも次の性質を追加する必要がある。

- 操作IDによる重複排除
- 点検票の版番号を使った条件付き更新
- 競合した変更の保留と確認キューへの登録
- 写真本体の保存完了後にメタデータを受理する処理
- 操作者、操作時刻、端末を含む変更イベントの追記
- 端末の有効性を確認し、失効済み端末を拒否する処理

既存APIを拡張するか、オフライン同期専用APIを設けるかは設計審査で決定する必要がある。同期では再送、部分成功、競合といったCRUD APIだけでは表現しにくい結果を扱うため、専用APIに分離する方が責務は明確になる。一方で、既存APIとの処理重複や移行コストが増える可能性がある。

## 9. 運用上の代償と障害時の扱い

この構成では、通信不能でも点検を継続でき、再送による重複や競合時の黙示的な上書きを防げる。その代わり、端末には業務データ、outbox、写真ファイル、個別の同期状態が存在し、オンライン専用アプリより状態管理が複雑になる。

写真の一時保存には端末容量が必要である。必要容量は最大200枚という枚数だけでは算出できないため、実際の写真サイズ、同期されない日数、再送用データの保持期間を確認して決める。容量不足が点検中に発生すると撮影を継続できなくなる可能性があるため、残容量の検査と不足時の扱いは実装前に定義する必要がある。

また、版不一致を自動上書きしないため、競合した変更の確定は即時ではない。確認キューが滞留した場合の担当者、優先順位、点検完了判定への影響を運用設計で定めなければならない。

障害解析では、写真送信、メタデータ送信、点検票更新、競合保留の各段階を識別できる記録が必要となる。ただし、監視項目、再試行回数、完了までの保証時間などの具体値は、現時点では定めない。

## 10. 設計審査で決める事項

本案を実装へ進めるため、設計審査では次の事項を決定する。

1. 既存CRUD APIを拡張するか、オフライン同期専用APIを新設するか。
2. 操作ID、写真ID、チェックサムをどの範囲で重複排除に使用するか。
3. 競合確認画面で比較・統合する単位と、解決権限を誰に与えるか。
4. 端末時刻とサーバー受信時刻を履歴上どのように扱うか。
5. 写真の想定サイズ、必要な端末容量、同期後のローカル削除条件をどう定めるか。
6. 失効確認を含む端末認証方式と、オフラインで利用を継続できる条件をどう定めるか。
7. 7年間の変更履歴について、検索、出力、保全、保存期間終了後の処理をどう実現するか。

これらが決まれば、通信不能時の継続利用、再送の安全性、競合の保全、長期監査という主要要件を保ったまま、API、データモデル、画面状態、運用手順の詳細設計へ進められる。
