# トンネル点検アプリのオフライン対応 設計案

## 目的と前提

トンネル内で最長8時間通信できない状況でも、点検員がAndroidタブレットで点検票の入力と写真撮影を継続でき、通信復帰後にデータを安全に同期できる構成とする。

対象は40人の点検員で、1台あたり1日最大200枚の写真を扱う。点検票は通常、作成した点検員が編集するが、後から監督者が訂正する場合がある。現行のオンライン前提CRUD APIとオブジェクトストレージは維持しつつ、端末側の保存・同期・競合処理を追加する。

## 基本方針

端末を、オフライン中も点検を完結できる一時的な作業場所として扱う。端末で行った変更は即時にローカルへ確定し、通信が利用可能になった時点でサーバーへ再送可能な操作として同期する。

写真と点検票メタデータは分けて扱う。写真本体を先にオブジェクトストレージへ送信し、成功を確認してから、その写真を参照する点検票のメタデータを送信する。これにより、サーバー上の点検票が未到達の写真を参照する状態を避ける。

同一の点検票が複数の端末または利用者によって変更された場合、サーバーは黙って上書きしない。版番号の不一致を競合として保留し、監督者の確認キューへ送る。

## 端末側の保存構成

端末には暗号化したSQLiteを置き、点検データと同期前の操作を保存する。写真本体は端末の暗号化領域にファイルとして保存し、SQLiteには写真ファイルのパスとチェックサムを保存する。

| 保存先 | 保持する内容 | 役割 |
|---|---|---|
| 暗号化SQLite | 点検票、写真メタデータ、サーバー版番号、outbox | オフライン作業と同期状態の管理 |
| 端末の暗号化領域 | 写真本体 | 大容量データの一時保存 |
| OSの安全な鍵領域 | 端末鍵 | 端末内データの保護 |

outboxには、サーバーへ未送信または送信結果が未確定の操作を記録する。点検票の作成・更新、写真メタデータの登録などを操作として保持し、アプリ終了や通信断の後でも再開できるようにする。

写真はファイルとして管理し、各ファイルにチェックサムを持たせる。同期前後に同じ写真であることを確認でき、再送時の重複判定にも利用できる。

## 同期の流れ

通信が使える状態になったら、バックグラウンド同期がoutboxを順に処理する。

1. 写真ファイルを、チェックサム付きでオブジェクトストレージへ送信する。
2. 写真の送信成功を確認する。
3. 写真への参照を含む点検票メタデータをCRUD APIへ送信する。
4. サーバーが受理した操作をoutboxから完了として扱う。

各処理は再実行しても重複しない設計とする。たとえば通信断により送信結果だけが端末へ戻らなかった場合でも、同じ写真や同じ操作を安全に再試行できなければならない。

写真送信が未完了の間は、その写真を参照するメタデータを送信しない。これにより、写真未到達による参照切れを防ぐ。一方、写真送信済みでもメタデータ送信前に端末が停止する可能性はあるため、同期状態はSQLiteで管理し、次回同期時にメタデータ送信を再開する。

## 点検票の更新と競合処理

サーバー上の点検票には版番号を持たせる。端末は取得時点の版番号を保持し、更新時にはその版番号を伴って送信する。

サーバーの版番号と端末が持つ版番号が一致する場合は更新を受理し、サーバーは新しい版番号を発行する。不一致の場合は更新を自動適用せず、競合として保留する。競合した変更は監督者の確認キューへ送り、監督者が内容を確認して訂正または採用を判断する。

この方式は、通常は一人の点検員が編集するという運用に適している。監督者による後日の訂正や、複数端末からの更新が重なった場合に、現場入力を失わせず、確認可能な状態で判断を後ろへ回せる。

代償として、競合はその場で解決されない。点検員の端末では送信済みでも、最終的な確定を待つ状態が発生しうるため、アプリ上で同期済み・保留中・競合中を区別して表示する必要がある。

## 変更履歴と監査

サーバーには、点検票に対する変更イベントを追記専用で保存する。各イベントには、少なくとも変更者、変更時刻、変更元の端末を記録する。

このイベント記録を7年間保存し、現在の点検票の状態とは別に、いつ誰がどの端末から変更したかを追跡できるようにする。訂正や競合解決も変更イベントとして残すことで、後からの経緯確認に利用できる。

追記専用の履歴は、通常のCRUD更新だけでは失われやすい変更過程を保持するためのものである。現在値の保存と履歴の保存を分けることで、点検票の利用性と監査性を両立する。

## 端末紛失への対応

端末鍵はOSの安全な鍵領域に保存する。アプリや通常のファイル領域から鍵を直接扱わないことで、端末に保存された点検データ、outbox、写真の保護を強める。

管理者は端末を失効できるようにする。紛失端末は以後の同期を認めず、必要に応じて再登録した端末で業務を継続する。端末失効時に、すでに端末内に存在するデータをどこまで遠隔で扱えるかは、ここでは決めない。OSと端末管理の機能を含めた運用判断が必要である。

## 主な設計上の代償

この構成では、通信断中の継続利用と安全な再同期を得る代わりに、端末状態と同期処理が複雑になる。特に、写真送信・メタデータ送信・再試行・競合保留の状態を一貫して管理しなければならない。

また、1台あたり最大200枚の写真を一時保存するため、端末容量を消費する。必要な保存容量、写真を端末から削除できる条件、容量不足時の業務上の扱いは、運用要件として別途決める必要がある。

競合を自動上書きしないため、監督者の確認が必要な変更は即時に確定しない。これは操作の速さよりも、点検記録を失わず監査可能に保つことを優先する判断である。

## 設計審査で確認したい事項

- 写真送信完了後にメタデータを送信する順序と、再実行可能な同期方式を採用するか。
- 版番号不一致を競合として保留し、監督者確認へ送る方針を採用するか。
- 変更イベントを追記専用で7年間保存する範囲と、監査記録として必要な項目を確定するか。
- 端末上の写真保存容量、削除条件、容量不足時の運用を定義するか。
- 端末失効後の扱いを、OSおよび端末管理の運用と合わせて決めるか。
