# トンネル点検アプリ オフライン対応設計案

## 1. 目的

Androidタブレットが最長8時間オフラインになる環境でも、点検票の編集と写真撮影を継続できるようにする。通信回復後はデータを安全に同期し、競合を黙って上書きせず、7年間追跡可能な変更履歴を残す。

本案は設計審査用の初期案であり、未決定の製品機能や性能保証値は含めない。

## 2. 設計方針

- 端末上のローカルデータを、オフライン中の作業元として扱う。
- ユーザー操作をoutboxに永続化してから同期する。
- 同期処理は途中で停止しても再開でき、何度実行しても重複登録しない。
- 写真は本体、メタデータの順に同期する。
- 点検票の競合は自動上書きせず、監督者が判断する。
- サーバー側の変更イベントは追記専用とし、監査証跡を7年間保持する。
- 端末紛失時の情報漏えいリスクを、暗号化、鍵管理、端末失効によって抑える。

## 3. 全体構成

```text
Androidタブレット
├─ 点検アプリ
├─ 暗号化SQLite
│  ├─ 点検票・項目
│  ├─ 写真メタデータ
│  ├─ ローカル同期状態
│  └─ outbox
├─ 暗号化ファイル領域
│  └─ 未送信・送信済み写真
└─ OSの安全な鍵領域
   └─ 端末暗号鍵

       通信回復後にバックグラウンド同期
                    │
                    ▼

サーバー
├─ 同期API
├─ 点検票DB（版番号付き）
├─ 追記専用変更イベント
├─ 競合確認キュー
├─ 端末・失効管理
└─ オブジェクトストレージ（写真）
```

現行CRUD APIはオンライン画面などで継続利用できるが、モバイル同期には、再送や競合を明示的に扱える同期用APIを追加するのが望ましい。

## 4. 端末側のデータ設計

### 4.1 点検データ

各レコードに少なくとも次を保持する。

- サーバー上のID
- 端末で生成した一意なID
- 点検票IDおよび項目ID
- 現在値
- 最後に取得したサーバー版番号
- ローカル更新日時
- 作業者ID
- 同期状態
  - 同期済み
  - 未送信
  - 送信中
  - 競合
  - エラー

IDは端末がオフラインでも生成でき、全端末間で衝突しない方式を使用する。

### 4.2 outbox

点検票の変更とoutboxへの操作追加は、同一SQLiteトランザクションで確定する。これにより、画面上は更新されたが同期対象に残っていない状態を防ぐ。

outboxの主な項目は以下とする。

| 項目 | 用途 |
|---|---|
| 操作ID | 冪等性を保証する一意なID |
| 対象ID | 点検票、項目、写真などの識別 |
| 操作種別 | 作成、更新、訂正、写真登録など |
| ペイロード | 変更内容 |
| 基準版番号 | 編集開始時に認識していたサーバー版 |
| 操作者・端末ID | 監査情報 |
| 発生日時 | 端末で操作した日時 |
| 状態・再試行回数 | 同期制御 |
| 最終エラー | 障害調査用 |

同期失敗時に操作内容を書き換えず、同じ操作IDで再送する。

### 4.3 写真

写真本体は端末の暗号化領域に保存し、SQLiteには次を保持する。

- 写真ID
- 対応する点検票・項目
- ローカルファイルパス
- チェックサム
- ファイルサイズ
- 撮影日時
- 撮影者・端末ID
- アップロード状態
- サーバー上のオブジェクト識別子

ファイルの書き込み完了後にチェックサムを計算し、メタデータとoutboxをトランザクション登録する。途中まで書かれた写真を同期対象にしない。

## 5. 同期処理

### 5.1 基本フロー

1. OSの制約、通信状態、認証状態を確認する。
2. outboxから未処理操作を取得する。
3. 写真をオブジェクトストレージへアップロードする。
4. サーバーまたはストレージ側でチェックサムを確認する。
5. 写真アップロード成功後、点検票と写真のメタデータを同期APIへ送信する。
6. サーバーが操作ID、版番号、結果を返す。
7. 端末側で同期済みまたは競合として記録する。
8. 必要に応じてサーバー側の最新変更を端末へ取り込む。

アプリ終了、通信断、OSによる停止がどの段階で起きても、未完了処理を再開できるようにする。

### 5.2 冪等性

- すべての操作に一意な操作IDを付ける。
- サーバーは処理済み操作IDと結果を記録する。
- 同じ操作IDが再送された場合、変更を再適用せず、以前の結果を返す。
- 写真は写真IDまたはチェックサムを含む安定したオブジェクトキーで識別する。
- 同一写真の再送時は、既存オブジェクトのチェックサムが一致すれば成功として扱う。
- メタデータ登録も操作IDによって重複を防ぐ。

チェックサムは通信破損の検出に使用する。異なる写真を同一写真とみなす要件があるかは未決定であり、チェックサムだけに業務上の同一性判定を担わせない。

### 5.3 順序と依存関係

写真メタデータは、対応する写真本体の保存成功が確認できるまでサーバー上で確定しない。

同じ点検票に対する複数操作には端末内の連番または依存関係を持たせ、順序を維持する。ある操作が失敗した場合、後続操作を無条件に先行適用しない。

異なる点検票の同期は独立して進められるため、一件のエラーで同期全体を停止させない。

### 5.4 再試行

通信障害やサーバーの一時障害は、待ち時間を段階的に延ばして再試行する。認証失敗、端末失効、入力不正、競合など、再試行で解決しない問題は自動再試行から分離する。

ユーザーには少なくとも以下を区別して表示する。

- 同期済み
- 未同期
- 同期中
- 競合確認待ち
- ユーザーまたは管理者の対応が必要
- 一時エラーで再試行待ち

## 6. 競合管理

### 6.1 検出

点検票にはサーバー版番号を持たせる。更新要求には、端末が編集の基準とした版番号を含める。

- 基準版と現行版が一致する場合：更新を適用し、版番号を進める。
- 一致しない場合：更新を適用せず、競合として保留する。

HTTPの成功応答として曖昧に処理せず、同期APIの結果で「適用済み」と「競合」を明確に区別する。

### 6.2 競合確認キュー

競合には以下の情報を保存する。

- サーバー上の現在値
- 端末から送られた変更内容
- 両者の版番号
- 操作者、端末、操作日時、受信日時
- 関連する写真
- 競合が発生した項目

監督者は差分を確認し、次のいずれかを選択する。

- サーバー側の値を採用
- 端末側の変更を採用
- 内容を編集して新しい値を採用

解決結果も新しい変更イベントとして保存する。元の変更や競合記録は削除・上書きしない。

初期実装では点検票単位の版管理が安全である。一方、別項目への独立した変更まで競合になる可能性があるため、実際の同時編集頻度を確認し、必要なら将来、項目単位の競合判定を検討する。

## 7. 変更履歴と監査

サーバーには業務データの現在状態とは別に、変更イベントを追記専用で保存する。

各イベントには少なくとも以下を記録する。

- イベントID
- 操作ID
- 対象と操作種別
- 変更前後または再現可能な差分
- 操作者
- 端末ID
- 端末上の操作日時
- サーバー受信日時
- 基準版と適用後の版
- 適用、競合、競合解決などの結果

端末時刻は誤差や改変の可能性があるため、監査ではサーバー受信日時も必ず保持する。

変更イベントと、監査上必要な写真・メタデータの保存期間は7年間とする。削除権限、訂正方法、バックアップ、復元確認、保存期間終了後の廃棄方法は別途運用設計が必要である。

## 8. セキュリティ

- SQLiteと写真ファイルを端末上で暗号化する。
- 暗号鍵はアプリ領域へ直接保存せず、OSの安全な鍵領域で管理する。
- 通信は暗号化し、端末と利用者をそれぞれ識別する。
- サーバーは端末ごとの有効・失効状態を管理する。
- 失効端末からの同期、認証更新、データ取得を拒否する。
- アプリ起動時と同期開始時に失効状態を確認する。
- 監督者の競合解決など、権限の高い操作にはサーバー側で認可を行う。
- ログへ点検内容、認証情報、写真パスなどを不要に出力しない。

端末失効は、オフライン中の端末内データを遠隔で即時消去できる保証ではない。そのため、OSの端末管理、画面ロック、暗号鍵の保護、必要に応じたリモートワイプを組み合わせる。失効確認の頻度や、長期間オフラインだった端末の利用制限は未決定事項とする。

## 9. 容量とデータ削除

最大撮影件数は全体で1日8,000枚である。

```text
40台 × 200枚／台／日 = 8,000枚／日
```

ただし、写真1枚当たりの容量、保持する撮影日数、再圧縮方針が未定のため、端末およびサーバーの必要容量は現時点では確定できない。

端末上の写真は、少なくとも次の条件をすべて確認してから削除候補にする。

- 写真本体のアップロードが完了している。
- チェックサムの一致が確認されている。
- メタデータがサーバーに確定している。
- サーバーから確定応答を受けている。
- 必要なローカル閲覧期間を経過している。

容量不足時に未送信写真を自動削除してはならない。撮影前の空き容量確認、警告、撮影停止条件については製品要件として決定する必要がある。

## 10. 現行システムへの主な変更

- モバイル向け同期APIの追加
- 操作IDによる冪等性管理
- 点検票への版番号追加
- 競合データと監督者向け確認キューの追加
- 追記専用変更イベントストアの追加
- 写真のチェックサム検証とアップロード状態管理
- 端末登録・失効管理
- 端末へ差分を配信する仕組み
- 同期状況、滞留、失敗を監視する運用機能

既存CRUD APIからの更新も、版管理と変更イベント記録を経由させる必要がある。同期APIだけに制御を入れると、既存経路からの更新が競合検出や監査を迂回してしまう。

## 11. 障害時の想定動作

| 状況 | 動作 |
|---|---|
| 写真送信中に通信断 | 同じ写真IDで再送する |
| 写真成功後、メタデータ送信前に停止 | 写真の存在とチェックサムを確認し、メタデータ送信から再開する |
| サーバー処理後、応答受信前に通信断 | 同じ操作IDを再送し、以前の処理結果を取得する |
| 点検票の版不一致 | 更新せず、競合確認キューへ送る |
| 一部の操作だけ不正 | 当該操作を保留し、他の独立した点検票は同期する |
| 認証期限切れ | 認証回復まで同期を停止し、ローカル操作は要件に従って継続する |
| 端末失効 | サーバーとの通信を拒否し、管理対象として扱う |
| 端末容量不足 | 警告し、未送信データを保護する。撮影継続可否は別途決定する |

## 12. テスト観点

- 8時間のオフライン作業後に全操作が同期されること
- 1台200枚の写真を含む同期
- 写真送信の各段階で通信を切断した場合の再開
- 同一操作や写真を繰り返し送っても重複しないこと
- 端末再起動、アプリ強制終了、OSによるバックグラウンド停止
- 複数端末および管理画面からの競合更新
- 競合解決後も元の変更履歴が残ること
- 端末時刻がずれている場合の監査記録
- 鍵が利用できない場合、端末失効時、認証期限切れ時の挙動
- SQLiteと写真ファイルの不整合検出
- ストレージ容量不足
- 7年間の履歴を想定した検索、バックアップ、復元
- 既存CRUD APIから更新した場合の版管理とイベント記録

## 13. 主な代償とリスク

- ローカル状態、outbox、写真、サーバー状態の組み合わせが増え、同期処理が複雑になる。
- 写真の一時保存に大きな端末容量が必要になる。
- 監督者による競合解決まで、変更の確定が遅れる。
- 点検票単位の版管理では、別項目の変更でも競合が発生する可能性がある。
- Androidのバックグラウンド実行制約により、通信回復後すぐに同期が始まるとは限らない。
- 追記専用イベントと写真の7年保存により、ストレージおよび運用コストが増加する。
- 端末失効だけでは、オフライン端末内のデータを即時消去できない。

## 14. 設計審査で決定が必要な事項

1. 写真の平均・最大サイズ、画質、圧縮方針
2. 端末に必要な最低空き容量と容量不足時の挙動
3. 同期開始条件と利用者による手動同期の要否
4. 点検票単位または項目単位の競合管理
5. 競合解決の権限、期限、通知方法
6. オフライン中の認証期限切れや端末失効への対応
7. サーバーから端末へ同期するデータの範囲
8. 写真および現在状態にも7年保存を適用するか
9. 端末上の送信済みデータを保持する期間
10. 同期完了時間、成功率、復旧時間などの保証値
11. 監査イベントの改ざん検知、バックアップ、復元要件
12. 既存CRUDクライアントを版管理へ移行する方法

初期実装では「確実に保存する」「重複させない」「競合を上書きしない」「履歴を失わない」を優先し、同期速度や競合の自動統合は運用実績を得てから段階的に最適化する。
