# Kite Mail APIで送信を予約する

Kite Mail APIでトランザクションメールを送信予約するには、`POST /v1/messages` を呼び出します。予約を受け付けると `202 Accepted` が返りますが、これはメールの配信完了を示すものではありません。

## エンドポイント

```text
POST https://api.kitemail.example/v1/messages
```

次のヘッダーは必須です。

```http
Authorization: Bearer <token>
Idempotency-Key: <一意なキー>
```

`Idempotency-Key` には、同じ送信処理を識別できる値を指定してください。

## リクエスト本文

本文には、少なくとも送信先、使用するテンプレート、テンプレートに渡す変数を指定します。予約日時を指定する場合は、`send_at` を追加します。

```json
{
  "to": "<宛先>",
  "template_id": "<テンプレートID>",
  "variables": "<テンプレート変数>",
  "send_at": "2026-07-15T10:00:00+09:00"
}
```

この例の値は説明用です。`to` と `variables` の具体的な型や構造は、利用するテンプレートとAPI仕様に合わせてください。

`send_at` にはRFC 3339形式の日時を指定します。指定できる日時は、リクエスト時点から7日以内です。1回のリクエストで指定できる宛先は100件までです。

## 成功時のレスポンス

受付に成功すると、APIは次のようなレスポンスを返します。

```http
202 Accepted
```

```json
{
  "message_id": "<メッセージID>",
  "status": "queued"
}
```

`status` が `queued` であることは、送信処理がキューに登録されたことを示します。メールが実際に配信されたことや、受信者に到達したことを意味しません。

## 冪等性キーの扱い

同じ `Idempotency-Key` と同じ本文で24時間以内に再送すると、最初のリクエストに対するレスポンスが返ります。通信エラーなどでレスポンスを受け取れなかった場合は、重複送信を避けるため、同じキーと同じ本文で再送してください。

同じ `Idempotency-Key` を使って本文を変更すると、APIは次のエラーを返します。

```http
409 Conflict
```

本文を変更して再送する場合は、新しい `Idempotency-Key` を発行してください。

## エラーと再試行

主なエラーと対応は次のとおりです。

| ステータス | 意味 | 対応 |
|---|---|---|
| `400` | 入力不正 | 本文や日時、宛先数を確認して修正する。そのまま再試行しない |
| `401` | 認証失敗 | `Authorization` とトークンを確認する。そのまま自動再試行しない |
| `409` | 冪等性キーの競合 | 同じキーで本文を変更していないか確認する。そのまま再試行しない |
| `429` | 送信頻度の上限超過 | `Retry-After` ヘッダーの指示に従って再試行する |

ネットワークエラーと `429` は再試行できます。再試行時は、同じ送信処理である限り同じ本文と `Idempotency-Key` を使用してください。

`400`、`401`、`409` を受け取った場合は、同じ内容をそのまま自動再試行しないでください。入力、認証情報、または冪等性キーの使い方を確認してから対応します。

## 実装時の確認事項

送信予約を実装する前に、次の点を確認してください。

- ベースURLが `https://api.kitemail.example` になっている
- メソッドとパスが `POST /v1/messages` になっている
- `Authorization: Bearer <token>` を付けている
- `Idempotency-Key` を付けている
- 宛先が1回あたり100件以内になっている
- `send_at` がRFC 3339形式で、現在から7日以内になっている
- `202 Accepted` を配信完了として扱っていない
- 通信エラーや `429` の再試行で、本文と冪等性キーを維持している
