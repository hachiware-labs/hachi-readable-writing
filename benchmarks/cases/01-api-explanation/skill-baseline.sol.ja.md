# Kite Mail APIでトランザクションメールを送信予約する

Kite Mail APIでは、`POST /v1/messages` に送信内容と送信日時を渡すことで、トランザクションメールを予約できます。このAPIが返す `202 Accepted` は予約を受け付けたことを示すもので、メールの配信完了を保証するものではありません。

## リクエスト

ベースURLとエンドポイントを組み合わせ、次のURLへリクエストします。

```text
POST https://api.kitemail.example/v1/messages
```

以下のヘッダーが必須です。

```http
Authorization: Bearer <token>
Idempotency-Key: <一意のキー>
Content-Type: application/json
```

`Idempotency-Key` は、ネットワークエラーなどで同じ処理を再試行したときに、メールが重複して予約されるのを防ぎます。送信操作ごとに一意の値を生成し、再試行時には最初のリクエストと同じキーを使用してください。

リクエスト本文には、次のフィールドを指定できます。

| フィールド | 内容 |
|---|---|
| `to` | 宛先。1回のリクエストにつき100件まで |
| `template_id` | 使用するテンプレートの識別子 |
| `variables` | テンプレートへ渡す変数 |
| `send_at` | 送信予定日時。任意。RFC 3339形式で、現在から7日以内 |

送信を予約する場合は、`send_at` を指定します。タイムゾーンの解釈違いを避けるため、UTCの `Z` またはUTCオフセットを含む日時を使用してください。

## リクエスト例

次の値は説明用の例です。実際のトークン、テンプレートID、宛先、日時に置き換えてください。

```bash
curl -X POST "https://api.kitemail.example/v1/messages" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Idempotency-Key: 2f45b98e-65ce-4ec7-bb71-89d00484d76a" \
  -H "Content-Type: application/json" \
  -d '{
    "to": [
      "user@example.com"
    ],
    "template_id": "order-confirmation",
    "variables": {
      "customer_name": "山田太郎",
      "order_number": "A-12345"
    },
    "send_at": "2026-07-15T01:00:00Z"
  }'
```

`send_at` は現在から7日以内でなければなりません。過去の日時や7日を超える日時を指定した場合の詳しい挙動は不明ですが、入力不正として扱われる可能性があるため、クライアント側でも範囲を検証してください。

## 正常に受け付けられた場合

予約が受け付けられると、APIは `202 Accepted` を返します。

```http
HTTP/1.1 202 Accepted
Content-Type: application/json
```

```json
{
  "message_id": "msg_example123",
  "status": "queued"
}
```

`status: queued` は、メッセージが送信待ちの状態になったことを表します。配信完了を意味しないため、`202` を受け取った時点で「メール送信済み」として扱わないでください。

この情報だけでは、予約後の状態確認方法や配信結果の取得方法は分かりません。必要な場合は、Kite Mail APIの追加仕様を確認してください。

## 重複送信を防ぐ仕組み

同じ `Idempotency-Key` と同じリクエスト本文を24時間以内に再送すると、APIは最初のレスポンスを返します。このため、応答を受け取れなかった場合でも、本文とキーを変えずに再試行すれば、同じメールが重複して予約されることを避けられます。

一方、同じキーを使いながら本文を変更すると、APIは `409 Conflict` を返します。送信内容を変更して新たに予約する場合は、新しい `Idempotency-Key` を発行してください。

## エラーと再試行

主なエラーへの対応は次のとおりです。

| ステータス | 意味 | 対応 |
|---|---|---|
| `400 Bad Request` | 入力が不正 | 本文、宛先数、`send_at` の形式や範囲などを修正する。自動再試行しない |
| `401 Unauthorized` | 認証に失敗 | トークンや認証ヘッダーを確認する。自動再試行しない |
| `409 Conflict` | 冪等性キーが別の本文と競合 | キーと本文の対応を確認する。自動再試行しない |
| `429 Too Many Requests` | 送信頻度の上限を超過 | `Retry-After` ヘッダーで指定された時間を待って再試行する |

ネットワークエラーでは再試行して構いません。再試行時は、`Idempotency-Key` とリクエスト本文を変更しないでください。本文を変更すると冪等性が成立せず、同じキーを使えば `409` になります。

`400`、`401`、`409` は、同じリクエストを繰り返しても解消しないため、そのまま自動再試行しません。原因を修正したうえで、必要に応じて新しい送信操作として実行してください。
