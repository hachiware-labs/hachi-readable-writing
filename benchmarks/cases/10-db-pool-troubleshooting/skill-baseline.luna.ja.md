# 本番API：データベース接続プール枯渇対応手順

## 目的

APIの `pool acquire timeout` が増加し、データベース接続プールの枯渇が疑われる場合に、当番エンジニアがサービスを安全に縮退・復旧させるための手順です。

この事象では、データベース自体のCPUやディスクが原因とは限りません。トランザクション中に外部決済APIを呼び出すことで、接続が2～8秒間保持され、プールの空きがなくなっている可能性があります。

## 1. 最初に行うこと

影響が継続している場合は、次の対応を直ちに開始します。

1. 障害の開始時刻と、対応開始時刻を記録する。
2. 可能であれば、切り戻し前に次の情報を保存する。

   - APIの稼働インスタンス数
   - 各インスタンスの接続プール上限
   - `pool acquire timeout` の発生状況
   - API p95
   - データベースの接続数と `idle in transaction` のセッション数
   - 現在のリリースバージョン

   実行コマンドや監視画面の名称は環境固有のため、既存の承認済みの手段で確認・保存します。証跡を取るために、障害対応中のセッションを先に終了したり、データベースを再起動したりしないでください。

3. 直前の問題リリースを切り戻す。
4. APIのインスタンス数を12台から8台へ一時的に減らす。
5. データベースの接続上限は変更しない。

各インスタンスのプール上限が20のままなら、8台で使用し得る接続数は理論上160です。12台のときの理論上240接続から減るため、データベース上限180との競合を緩和できます。

## 2. 事象の見立て

現在分かっている関係は次のとおりです。

- APIは12台で動作し、各インスタンスのプール上限は20です。
- API側では理論上240接続を確保できますが、データベースの上限は180接続です。
- 障害時にはAPI p95が300ミリ秒から9秒へ悪化し、`pool acquire timeout` が増加しています。
- データベースCPUは45%以下で、ディスク待ちと遅いSQLの増加もありません。
- 直前のリリースで、外部決済APIの呼び出しがデータベーストランザクション内に移されています。
- 外部決済APIの応答には2～8秒かかります。
- `idle in transaction` のセッションが増加しています。

そのため、現時点ではデータベースの処理能力不足よりも、外部APIの応答待ちの間もデータベース接続を保持し続けていることが主因と考えます。

概念的には、次の順で接続が塞がります。

```text
トランザクション開始
  ↓
外部決済APIを呼び出す
  ↓ 2～8秒待機
データベース接続とトランザクションを保持
  ↓
プールの空き接続が減少
  ↓
新しいリクエストが接続取得待ちになる
  ↓
pool acquire timeout とAPI p95悪化
```

## 3. 切り戻し後の確認

切り戻しと8台への縮退が完了したら、次の状態を確認します。

- APIの稼働インスタンス数が8台になっている。
- 問題のリリースから切り戻されている。
- APIの `pool acquire timeout` が減少し、発生し続けていない。
- API p95が低下傾向にある。
- `idle in transaction` が減少傾向にある。
- データベース接続数が上限180に張り付いていない。
- エラー増加やリクエスト滞留が新たに発生していない。

一部の指標だけが改善している場合は、復旧とは判断しません。`pool acquire timeout` が継続している場合は、APIの接続プール状況とデータベースのセッション状態を再確認し、インスタンス数を増やさないでください。

## 4. 復旧判定

次の条件をすべて満たした状態を、障害からの復旧とします。

- 30分間、`pool acquire timeout` が発生していない。
- API p95が500ミリ秒未満である。
- `idle in transaction` が平常値に戻っている。

条件を満たすまでは、8台への一時縮退を維持します。いずれかの条件が再び崩れた場合は、復旧判定を取り消し、障害対応を再開します。

## 5. 復旧後に行う調査

サービスが安定した後、次の順で原因を確認します。

1. 問題のリリース差分を確認し、外部決済APIの呼び出しがトランザクション内へ移された経緯を特定する。
2. 対象処理のトランザクション境界を確認する。
3. 外部決済APIの呼び出し中に、データベース接続とトランザクションが保持されていないか確認する。
4. 外部APIの応答時間と、接続保持時間を比較する。
5. 障害発生時の `idle in transaction` の増加と、`pool acquire timeout` の増加が同じ時間帯に起きているか確認する。
6. API以外のサービスがデータベース接続を使用していないか確認する。

調査中も、データベースの接続上限を引き上げて現象を隠す対応は行いません。接続上限の変更が必要な場合は、別途、データベース全体の利用者と容量を確認したうえで判断します。

## 6. 恒久対応

対象処理は、次の構造に変更します。

```text
短いトランザクション開始
  ↓
更新に必要な情報を読み取る
  ↓
トランザクション終了
  ↓
外部決済APIを呼び出す
  ↓
結果を反映するための短いトランザクション開始
  ↓
必要な更新を行う
  ↓
トランザクション終了
```

外部決済APIの呼び出しはトランザクションの外へ移し、データベース接続を保持する時間を、読み取りや更新に必要な短い区間だけにします。

あわせて、各インスタンスの接続プール上限を、データベースの総接続上限から逆算して設定します。設定値は、APIの全インスタンス数だけでなく、他サービスの接続数、管理用接続、障害時の余裕を含めて決めます。各インスタンスの上限を単純に20へ固定するのではなく、全インスタンスの合計がデータベース上限180を超えない構成にします。

恒久対応の完了後は、通常のインスタンス数へ戻す前に、少なくとも次を確認します。

- 外部決済APIの待機中にデータベース接続を保持しない。
- 通常のインスタンス数で、接続プールの合計上限がデータベースの許容範囲内に収まる。
- 負荷試験または同等の検証で、外部APIが2～8秒かかっても接続プールが枯渇しない。
- `idle in transaction` と `pool acquire timeout` を監視できる。
- 復旧条件を30分間満たすことを確認できる。

## 7. 連絡と対応終了

対応中は、リリース担当、API担当、データベース担当に、次の内容を共有します。

- 影響開始時刻と現在の影響
- 問題リリースを切り戻したこと
- APIを8台へ縮退したこと
- データベース接続上限を変更していないこと
- 復旧条件の達成状況
- 恒久対応の担当者と期限

`pool acquire timeout` が30分間発生せず、p95が500ミリ秒未満で、`idle in transaction` が平常値へ戻ったことを確認できたら、応急対応を完了します。その後、原因調査と恒久対応の追跡に切り替えます。
