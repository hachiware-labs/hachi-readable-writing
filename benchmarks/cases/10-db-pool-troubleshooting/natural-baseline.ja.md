# 本番API データベース接続プール枯渇 対応手順書

## 目的

本番APIで `pool acquire timeout` が増加し、応答時間が悪化した際に、当番エンジニアが安全に切り分け、応急対応、復旧判定、引き継ぎを行うための手順書です。

## 事象の概要

- API：12インスタンス
- 1インスタンスあたりの接続プール上限：20
- API全体の理論最大接続数：240
- データベースの接続上限：180
- 障害時のAPI p95：300ms → 9秒
- `pool acquire timeout` が増加
- データベースCPU：45%以下
- ディスク待ち：通常範囲
- 遅いSQL：増加なし
- `idle in transaction` セッション：増加
- 直前のリリースで、2～8秒かかる外部決済API呼び出しがデータベーストランザクション内へ移動

## 現時点の原因判断

第一原因は、外部決済APIの応答待ちの間もトランザクションとデータベース接続を保持していることです。

接続の保持時間が2～8秒まで延び、利用可能なプール接続が減少した結果、後続リクエストが接続取得待ちとなり、`pool acquire timeout` とAPI遅延が発生しています。

また、API側では最大240接続を要求できる一方、データベース上限は180接続です。この不整合が枯渇を悪化させます。

データベースCPU、ディスク待ち、遅いSQLに異常がないため、現時点ではデータベースの処理能力不足やSQL性能劣化を主因とは判断しません。

# 障害対応手順

## 1. 事象を確認する

次の状態を確認します。

- `pool acquire timeout` が平常時より増加している
- APIのp95が悪化している
- `idle in transaction` が増加している
- データベースCPU、ディスク待ち、遅いSQLに顕著な悪化がない
- 障害開始時刻が問題のリリース以降である

監視画面や実行コマンドの名称は未確定です。組織の標準監視およびデプロイ手順を使用してください。

確認時に以下を記録します。

- 障害を認知した時刻
- API p95
- `pool acquire timeout` の件数または発生率
- APIインスタンス数
- データベース接続数と接続上限
- `idle in transaction` の件数
- データベースCPUとディスク待ち
- 適用中のリリース識別子
- 問題のリリース直前の正常なリリース識別子

## 2. インシデントを宣言する

利用者影響が発生しているため、所定のインシデント連絡先へ次を共有します。

> 本番APIでデータベース接続プールの枯渇が発生しています。API p95が約9秒まで悪化し、`pool acquire timeout` が増加しています。直前のリリースによる長時間のトランザクション保持が原因と判断し、リリースの切り戻しとAPIインスタンス数の一時削減を実施します。データベース接続上限は変更しません。

## 3. 問題のリリースを切り戻す

直前の正常なリリースへ切り戻します。

- 組織の標準デプロイ／ロールバック手順を使用する
- 切り戻す対象とリリース識別子を作業者同士で確認する
- データベーススキーマやデータに後方互換性の問題がないことを確認する
- 作業開始時刻と完了時刻を記録する
- 切り戻し後、新しいAPIエラーが増えていないことを確認する

コマンドが不明なため、本書の情報だけで推測したコマンドを実行しないでください。標準手順が見つからない場合は、デプロイ責任者へエスカレーションします。

## 4. APIインスタンス数を8へ減らす

切り戻し後、APIインスタンス数を12から8へ一時的に減らします。

8インスタンス × 20接続 = 最大160接続となり、データベース上限180を下回ります。残り20接続はAPI以外の利用や運用接続の余裕として扱います。

注意事項：

- インスタンス削減によりAPI処理能力が低下するため、リクエスト数、CPU、メモリ、エラー率も監視する
- 一度に変更することが標準手順上危険な場合は段階的に減らす
- 終了対象への新規トラフィックが停止し、処理中リクエストが排出される手順を使用する
- データベース接続上限は変更しない
- 接続を空ける目的で、データベースや全APIインスタンスを一斉再起動しない

## 5. 回復傾向を確認する

切り戻しとインスタンス削減後、次を継続して監視します。

- `pool acquire timeout`
- API p95
- APIエラー率
- APIのリクエスト数、CPU、メモリ
- データベースの総接続数
- `idle in transaction`
- データベースCPU、ディスク待ち
- 外部決済処理の成功率とエラー率

`idle in transaction` は、既存セッションが終了するまで段階的に減少する可能性があります。減少傾向が見られるか確認します。

# 復旧判定

次のすべてを連続30分間満たした場合に復旧と判定します。

- `pool acquire timeout` が0件
- API p95が500ms未満
- `idle in transaction` が平常値に復帰

あわせて、次の悪化がないことも確認します。

- APIエラー率の増加
- インスタンス削減によるCPUまたはメモリの逼迫
- 外部決済処理の失敗増加
- データベース接続数の再上昇

30分の計測開始時刻と復旧判定時刻を記録します。

復旧連絡例：

> 本番APIのデータベース接続プール枯渇は復旧しました。問題のリリースを切り戻し、APIを一時的に8インスタンスへ変更しています。30分間 `pool acquire timeout` は発生せず、API p95は500ms未満、`idle in transaction` も平常値に戻っています。恒久対応が完了するまで、現在の構成と監視を継続します。

---

# 回復しない場合

切り戻しとインスタンス削減後も改善しない場合は、次を確認してデータベース担当およびアプリケーション担当へエスカレーションします。

- 切り戻しが全インスタンスへ反映されたか
- 問題のバージョンが残存していないか
- 長時間の `idle in transaction` セッションが残っていないか
- API以外のクライアントが接続数を消費していないか
- 接続リークや異常に長いトランザクションが別経路にもないか
- 外部決済APIの遅延や失敗が継続していないか
- データベース接続数が上限付近に張り付いていないか

セッションの強制終了は、未完了トランザクションのロールバックや利用者処理への影響を伴います。対象、影響、終了基準が確認できない状態では実施せず、データベース担当の判断を仰ぎます。

インスタンス数をさらに減らすと、API処理能力が不足する可能性があります。追加削減は自動的に行わず、トラフィック量とAPI負荷を踏まえてインシデント責任者が判断します。

# 恒久対応

## アプリケーション修正

外部決済APIの呼び出しをデータベーストランザクション外へ移します。

処理の基本形は次のとおりです。

1. トランザクション外で外部決済APIを呼び出す
2. 応答を検証する
3. データベース更新に必要な短い区間だけトランザクションを開始する
4. 必要な更新を行い、直ちにコミットまたはロールバックする

実装時は、外部APIの再試行による二重決済を防ぐため、冪等性キー、処理状態管理、重複応答への対応も確認します。

## 接続プールの再設計

各インスタンスのプール上限は、単純にデータベース上限をインスタンス数で割るのではなく、API以外の接続と運用余力を差し引いて決めます。

```text
1インスタンスのプール上限
  <= floor(
       (DB接続上限 - API以外の最大接続数 - 運用予備接続数)
       / APIの最大インスタンス数
     )
```

現在の12インスタンスでプール上限20の設定は、最大240接続となるため使用できません。API以外の接続数と必要な予備枠を確認したうえで、新しい上限を決定します。

## 再リリース条件

恒久対応版の再リリース前に、少なくとも次を検証します。

- 外部API待ちの間、データベース接続とトランザクションを保持しない
- 外部APIが8秒以上遅延してもプール枯渇が起きない
- 想定最大インスタンス数でも全体の接続予算を超えない
- タイムアウト、再試行、二重決済防止が期待どおり動作する
- 負荷試験で `pool acquire timeout` が発生しない
- `idle in transaction` が平常範囲を維持する

恒久対応の展開と監視完了後に、APIインスタンス数を通常値へ戻します。先に12インスタンスへ戻す場合は、接続プール上限の修正が反映済みで、全体の接続予算を超えないことを必須条件とします。
