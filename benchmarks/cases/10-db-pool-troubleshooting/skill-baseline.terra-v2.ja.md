# データベース接続プール枯渇 障害対応手順

`pool acquire timeout` の増加、API p95 の大幅な悪化、または `idle in transaction` の増加を確認した場合に実施する。本障害では、外部決済 API 呼び出しがトランザクション内で長時間接続を保持し、API 側の接続需要がデータベース上限を超え得ることが主因と考える。

## まず行うこと

1. 問題のリリースを切り戻す。外部決済 API 呼び出しをトランザクション内へ移した変更を含むリリースが対象である。

2. API インスタンス数を一時的に 8 へ減らす。

   各インスタンスの接続プール上限は 20 のため、12 インスタンスでは理論上 240 接続となる。データベースの接続上限は 180 であり、全プールが使用されると上限を超える。8 インスタンスへ減らすと理論上の上限は 160 接続になる。

3. データベースの接続上限は変更しない。

   今回はデータベース CPU が 45% 以下、ディスク待ちも通常範囲で、遅い SQL の増加も確認されていない。接続上限の引き上げは、長時間トランザクションによる接続占有を残したまま負荷を広げるだけになるため、応急対応として行わない。

## 状況確認

監視画面や利用可能なログで、次を確認する。

- `pool acquire timeout` の発生数
- API の p95 レイテンシ
- 接続中セッションにおける `idle in transaction` の件数
- データベースの接続数
- データベース CPU、ディスク待ち、遅い SQL の状況

今回の典型的な症状は、API p95 が通常の約 300 ミリ秒から約 9 秒へ悪化し、`pool acquire timeout` が増える一方、データベース CPU・ディスク待ち・遅い SQL には大きな悪化がない状態である。

## 原因の判断

直前のリリースで、外部決済 API の呼び出しがデータベーストランザクション内へ移された。外部 API は 2～8 秒かかるため、その待機中も接続とトランザクションが保持される。結果として `idle in transaction` が増え、空き接続を待つ API リクエストが `pool acquire timeout` になる。

CPU 使用率や SQL 実行時間ではなく、接続を保持する時間が問題である点に注意する。

## 復旧判定

切り戻しとインスタンス数の調整後、次の状態が連続 30 分間続けば復旧と判断する。

- `pool acquire timeout` が発生しない
- API p95 が 500 ミリ秒未満である
- `idle in transaction` が平常値に戻っている

条件を満たさない場合は、切り戻しが反映されていること、API インスタンス数が 8 であること、長時間の `idle in transaction` セッションが継続していないことを確認し、担当者へエスカレーションする。

## 恒久対応

外部決済 API の呼び出しをトランザクション外へ戻す。データベース更新に必要な短い区間だけをトランザクションにし、外部 API の待機中に接続を保持しない設計へ変更する。

あわせて、各 API インスタンスの接続プール上限は、データベース接続上限から逆算して設定する。インスタンス数とプール上限の積がデータベース接続上限を超えないようにし、運用上必要な余裕も確保する。
