# 本番API：データベース接続プール枯渇 障害対応手順

## 目的

`pool acquire timeout` の増加とAPI遅延が発生したとき、接続を長時間占有している処理を止め、サービスを安全に復旧させる。

## 直ちに行うこと

1. 現在のリリース識別子、APIのp95、`pool acquire timeout` 件数、`idle in transaction` セッション数を記録する。監視画面名・実行コマンドは未整備のため、手元で利用可能な監視・ログ・DBセッション確認手段を使う。
2. 問題のリリースを切り戻す。
3. APIインスタンス数を一時的に **12台から8台へ減らす**。
4. データベースの接続上限は変更しない。

切り戻しや台数変更の前に、原因特定に必要な最低限の記録だけを残す。調査のために復旧を遅らせない。

## 発動の目安

次の状況では、この手順を開始する。

- APIのp95が平常時の約300ミリ秒から大きく悪化し、秒単位になっている
- `pool acquire timeout` が継続して発生または増加している
- DBセッションで `idle in transaction` が増えている
- DB CPUが45%以下、ディスク待ちが通常範囲で、遅いSQLの増加もない

この組み合わせでは、DB自体の処理能力不足よりも、アプリケーションが接続を返却できずに待機列を作っている可能性が高い。

## 判断と対応

今回のAPIは、1インスタンスあたり最大20接続、12インスタンスで理論上最大240接続を要求し得る。一方、DBの接続上限は180である。さらに、直前のリリースでは外部決済API呼び出しがトランザクション内へ移されており、外部API応答に2〜8秒かかる。その待機中もDB接続が保持されるため、`idle in transaction` が増え、空き接続が枯渇する。

| 観測結果 | 判断 | 対応 |
|---|---|---|
| `pool acquire timeout` 増加、p95悪化、`idle in transaction` 増加 | 接続がトランザクション内で長時間保持されている可能性が高い | 切り戻しと8台への縮退を実施する |
| DB CPU・ディスク待ち・遅いSQLに異常がある | 本手順だけでは原因を限定できない | DB負荷・SQL障害として並行調査する |
| 切り戻し後もタイムアウトが増える | 保持済み接続の解消待ち、または別要因の可能性 | セッション数・エラー率・直近変更を確認し、障害担当へエスカレーションする |

## 復旧確認

切り戻しと縮退後、以下を連続30分確認する。

- `pool acquire timeout` が発生しない
- API p95が500ミリ秒未満である
- `idle in transaction` が平常値に戻る

3条件を満たした時点で復旧と判断する。満たさない場合は、DBの接続上限を変更せず、残存する `idle in transaction` セッション、切り戻しの反映状況、追加リリースやトラフィック変化を確認する。

## 恒久対応

外部決済APIの呼び出しをDBトランザクション外へ戻す。DB更新が必要な短い区間だけトランザクションを開始・終了し、外部APIの応答待ち中にDB接続を保持しない。

あわせて、各APIインスタンスの接続プール上限は、DB全体の接続上限から逆算して設定する。全インスタンスの理論上限がDB上限180を超えないようにし、DB運用や他コンポーネント用の接続枠も考慮して余裕を持たせる。
