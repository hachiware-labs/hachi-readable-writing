# 本番API：データベース接続プール枯渇 対応手順書

## 1. 事象

APIのp95レイテンシが通常の約300ミリ秒から9秒へ悪化し、`pool acquire timeout` が増加している。

データベースCPU、ディスク待ち、遅いSQLの増加は確認されていない。一方で、接続セッションに `idle in transaction` が増加している。

## 2. まず行う対応

1. 新規リリースや設定変更を停止する。
2. 直前のリリースを切り戻す。
3. APIインスタンス数を12台から8台へ一時的に減らす。
4. データベースの接続上限は変更しない。
5. 切り戻し後、以下の復旧条件を確認する。

## 3. 状況の見立て

各インスタンスの接続プール上限は20であり、12インスタンスでは理論上最大240接続となる。

```text
12インスタンス × 20接続 = 最大240接続
データベース上限             = 180接続
```

アプリケーション側の理論最大接続数が、データベースの接続上限を超えている。

さらに直前のリリースで、外部決済APIの呼び出しがデータベーストランザクション内へ移された。外部APIは2〜8秒かかるため、その間トランザクションと接続が保持され、他のリクエストが接続を取得できず待機している可能性が高い。

`idle in transaction` の増加は、この仮説を裏付ける兆候である。

データベースCPUが低いことは、接続プール枯渇を否定しない。今回はSQL処理能力ではなく、接続が長時間保持されていることが主な問題と考える。

### 4.2 APIインスタンス数を8台へ減らす

```text
[APIインスタンス数を12台から8台へ変更する手順・コマンドを記載]
```

8台の場合、接続プール上限20のままでも理論上は次の値になる。

```text
8インスタンス × 20接続 = 最大160接続
```

データベース上限180を下回るが、他のアプリケーションや管理接続が使用する分もあるため、接続数が十分に余っていることを確認する。

確認項目：

- APIインスタンス数が8台になっていること
- 各インスタンスのプール設定が想定どおりであること
- データベース接続数が上限に張り付いていないこと
- `pool acquire timeout` が減少または停止していること
- APIのp95が改善していること

## 5. 復旧判定

以下をすべて満たした状態を30分間維持する。

- `pool acquire timeout` が発生しない
- API p95が500ミリ秒未満
- `idle in transaction` が平常値へ戻る
- APIのエラー率が平常値
- データベース接続数が上限付近で推移していない

## 7. 恒久対応

外部決済APIの呼び出しをデータベーストランザクション外へ移動する。

推奨される処理順序は次のとおり。

1. 必要な状態を短いトランザクションで読み取る。
2. トランザクションを終了する。
3. 外部決済APIを呼び出す。
4. 結果を短い別トランザクションで更新する。
5. 失敗時の再試行、重複実行防止、状態不整合への対策を実装する。

外部API呼び出し中にデータベース接続やトランザクションを保持しないことを、コードレビューとテストで確認する。

## 8. 接続プール設定の見直し

接続プール上限は、データベース全体の接続上限から逆算する。

考慮する値：

```text
APIのインスタンス数
× 1インスタンスあたりのプール上限
+ 他サービス・管理接続の必要数
< データベース接続上限
```

余裕を確保した上で、APIのインスタンス増加時にもデータベース上限を超えない設定にする。設定変更後は、負荷試験で接続数、待機時間、トランザクション保持時間を確認する。
